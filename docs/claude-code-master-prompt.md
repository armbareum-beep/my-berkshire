# ENUF — Claude Code 마스터 프롬프트

> **사용법:** 이 파일 전체를 Claude Code에 붙여넣지 말고, **STEP 0부터 한 단계씩** 붙여넣어라. 각 STEP이 끝나고 동작을 확인한 뒤 다음 STEP으로 넘어가야 Claude Code가 엉키지 않는다.
>
> **사전 준비 (컴퓨터에서 1회):**
> 1. 새 폴더 만들고 그 안에서 Claude Code 실행.
> 2. `rational-capital-prd-v0.7.md`, `xirr-spec-v1.md`, `user-rails-v1.md`, `design-strategy-v1.md`, `lookthrough-financials-spec-v1.md` 다섯 파일을 그 폴더의 `/docs` 안에 넣어둔다. (Claude Code가 읽을 단일 진실원본이다. 단 lookthrough는 V2 엔진이라 V0~V1 단계에선 참고만.)
> 3. Supabase 대시보드 → Settings → API 에서 `Project URL`, `anon key`, `service_role key` 를 복사해 `.env.local` 에 넣는다. **이 키들은 절대 프롬프트나 코드에 직접 박지 않는다.**

## STEP 1 — DB 스키마 + RLS

```
STEP 1만 수행하라. /docs/rational-capital-prd-v0.7.md 의 "부록: 백엔드 DB 스키마 설계 프롬프트" 섹션을 그대로 구현하라. 핵심만 다시 못 박는다:

- 테이블: holdings, accounts, events. (cash_flows/transactions 분리 금지 — 단일 events 테이블, type enum: BUY/SELL/DIVIDEND/DEPOSIT/WITHDRAWAL)
- events: type, symbol(nullable), quantity(nullable), price_or_amount, fee_and_tax(numeric default 0), date(필수), account_id FK
- accounts: account_type enum(GENERAL/ISA/PENSION/OVERSEAS), commission_rate(numeric), holding_id FK
- holdings: founded_at, initial_capital, initial_valuation (설립 등기 1회성), mode enum('ledger','challenge','live') — 데이터 검증 수준·경쟁 범위 결정. v0은 ledger·challenge만 사용, live(라이브)는 V3. 한 사용자가 mode가 다른 holdings를 여러 개 보유 가능(모드는 회사 단위 고정, 전환 불가).
- 가입 시 holding마다 기본 account('main', GENERAL) 자동 생성(트리거 또는 가입 로직).
- positions: 매수/매도 events에서 계좌·종목별 수량·평균단가를 계산하는 VIEW. 사용자 직접 수정 불가.
- RLS: 사용자는 자기 holdings/accounts/events만 read/write.
- 세율은 events에 저장하지 않고 lib/config 상수로. 실시간 시세도 저장 안 함.
- Supabase migration SQL 로 작성하고, 그에 맞는 TypeScript 타입 생성.

끝나면 멈추고, 마이그레이션을 어떻게 적용하는지(명령) 알려달라. 요약 3줄.
```

## STEP 2 — XIRR 엔진 + 테스트 (제품의 심장, 가장 중요)

```
STEP 2만 수행하라. /docs/xirr-spec-v1.md 를 한 줄도 빠짐없이 그대로 구현하라.

- lib/finance/xirr.ts 에 xirr(flows) 구현: Newton-Raphson → 실패 시 Bisection 폴백.
- 현금흐름 정의를 명세대로: 설립평가액(−), 입금(−), 출금(+), 오늘 평가액(+). 배당·매수·매도는 현금흐름에 직접 넣지 말 것(평가액 통해 반영).
- 현재 평가액 = Σ(보유수량 × 시세) + 현금잔고. 현금잔고 = 입금 − 출금 + 배당 + 매도대금 − 매수대금 − Σ(fee_and_tax).
- 엣지케이스 6개 전부 처리(특히 설립 90일 미만은 연환산 금지 → 누적수익률 표시).
- CAGR 보조지표도 명세대로.
- **명세 7번의 테스트 케이스 A/B/C를 자동 테스트로 작성하고 반드시 통과시켜라.** 통과 못 하면 구현이 틀린 것이다. 테스트 통과 결과를 보여줘라.

이 STEP에서는 시세를 목업으로(고정값) 둔다. 끝나면 멈추고 테스트 결과 + 요약 3줄.
```

## STEP 3 — 인증 + 첫 5분 강제 여정 (설립등기 → 첫 매수 → 첫 XIRR)

```
STEP 3만 수행하라. /docs/user-rails-v1.md 의 1번 "첫 5분 강제 여정" J0~J5 표를 그대로 구현한다.

- Supabase Auth(이메일 매직링크 또는 OAuth)로 로그인/회원가입.
- **로그인 라우팅(레일 0-1):** 로그인 직후 사용자의 holding 보유 여부로 분기. Holding 있으면 대시보드 직행(온보딩 건너뜀), 없으면 J0부터. 매 로그인마다 온보딩을 반복시키지 말 것.
- **J0 모드 선택:** 가입 직후 "어떻게 시작할까요?" → [챌린지로 시작](challenge: 돈 없이 오늘부터, 남과 경쟁) 또는 [장부로 시작](ledger: 과거 포함 내 기록, 나·시장과 경쟁). 선택값을 holdings.mode에 저장. 라이브(live, 연동)는 "준비 중"으로 흐리게 노출(V3). "나중에 다른 회사도 설립 가능" 안내 한 줄.
- 이어서 J1(회사명) → J2(설립일·초기자본금·보유종목 등기) → J2.5(설립의식) → J3(첫 인수 제안) → J4(매수 기록) → J5(대시보드 착지 + XIRR 카드 1회성 하이라이트)를 끊김 없는 단방향 레일로. 중간에 메뉴/대시보드로 샐 수 없다.
- J3의 "나중에 하기"는 작은 텍스트 링크로만(버튼 아님). 누르면 대시보드 + 미완료 꼬리표 띠(레일 문서 4번).
- 버튼 문구는 표의 문구를 그대로 쓴다([회사 설립하기], [등기 완료], [첫 인수 기록하기], [인수 체결]).
- 등기 완료 시 holdings + 기본 account 생성. 설립 등기 이전 기간 수익률은 계산하지 않는다.
- 90일 미만이면 XIRR 대신 누적수익률 + "연환산 수익률은 설립 90일 후 공개됩니다."
끝나면 멈추고 요약 3줄.
```

## STEP 4 — 투자 이벤트 기록 + Next Action 엔진

```
STEP 4만 수행하라.
- 다섯 버튼: 인수(BUY)·매각(SELL)·배당(DIVIDEND)·증자(DEPOSIT)·인출(WITHDRAWAL). 모든 입력에 날짜 필수.
- 종목 검색: 입력 시 토스증권식 행(원형 로고 + 종목명 + 티커)으로 결과 표시. 선택된 종목도 로고+티커로 확인되게. 로고는 티커로 시세 API 조회, 폴백은 이니셜 원형.
- 수수료·세금: fee_and_tax 는 선택 입력. 비우면 계좌 commission_rate + 계좌유형 세율(v0는 GENERAL: 배당 15.4% + 거래세)로 자동 추정해 채운다.
- 미래 날짜 거부, 출금이 현금잔고 초과 시 거부(명세 엣지케이스).
- **거래일 모드 규칙(레일 5-2):** mode가 challenge·live면 거래일=오늘로 고정(S4 날짜 선택 건너뜀, 소급 불가 → 순위 조작 차단). mode가 ledger면 S4에서 과거 날짜 선택 허용.
- **이벤트 수정·삭제(PRD 4):** ledger는 자유 수정·삭제. challenge·live는 **입력 당일은 자유 수정·삭제**, 날짜가 지난 과거 이벤트는 직접 수정·삭제 금지·오늘 날짜 상쇄(reversing)만 허용. 삭제/취소 이력은 보존(감사 추적).
- 기록 즉시 "체결" 피드백(가벼운 모션). 버튼과 완료 토스트는 같은 동사([인수 체결] → "체결되었습니다").

- **Next Action 엔진:** /docs/user-rails-v1.md 2번 표를 getNextAction(userState) 단일 함수로 구현. 우선순위 1~5를 위에서부터 검사해 처음 맞는 것 하나만 반환. 대시보드 최상단 버튼은 항상 이 함수 결과 정확히 1개. 화면들이 각자 판단하지 말 것.

    - **리밸런싱 계획 저장 + 체크리스트(다음 작업 — 배당 앞):** "얼마 넣으면 어디에 살까" 계산 결과를 **자본배분 계획**으로 저장(holdings.active_plan jsonb: createdAt + legs[{symbol,name,shares}], 1개 활성). 진행률은 **이벤트에서 자동 파생**: 저장 시점 누적 매수 주수를 **기준선(baseBought)**으로 박고 이후 *증가분* ≥ 계획 주수면 체결 ✅(별도 done 상태 없음). **날짜 비교 금지** — 챌린지는 모든 매수가 "오늘"이라 기존 보유분과 구분 불가 → 기준선 방식 필수. 표시: 종목별/국가별/유형별 **모든 리밸런싱 탭 상단** + 대시보드 배너("계획 2/4 종목 체결 →"), 미체결은 딥링크로 바로 인수(`?type=BUY&symbol=&qty=`, 구현됨). 게이미피케이션=진행바/미션, 브랜드=자본배분 규율. 알림 리마인더는 STEP 6 알림에 묶음. **선행 없음(기존 이벤트만). 순서: 리밸런싱 계획 → 배당(B) → 증권사(A).** 카테고리 계산기에서도 종목별 주수+딥링크 표시(구현됨).

  - **다자산 + 부채 = 자본배분 확장 (큰 마일스톤 — 브랜드 핵심):** 지주회사 = 자본배분가. 주식뿐 아니라 **금·부동산·상가·코인**에 배분하고 **레버리지(대출)**를 쓰느냐가 자본배분의 핵심 의사결정(버크셔도 플로트로 다자산 배분). **프레이밍 원칙: "자본을 어디에 배분했나 + 레버리지 리스크" 렌즈(능동 배분가)여야 함 — 수동 가계부 ❌.** 이 프레이밍만 지키면 line 197(풀 가계부 복제 금지)과 충돌 안 함(line 197은 *수동 트래킹* 경계).

(딥링크 `/networth?add=debt`도 등록 폼 자동 열림+스크롤로 유지.) **V1 = 현재 잔액 수기 모델**(상환·이자 자동화 없음). **결정: 대출 기록이 현금을 자동 생성하지 않음**(부채만 등록) — 대출금은 대개 이미 자산(집·주식)에 들어가 있어 자동 현금화하면 이중계상되고 XIRR 오염(빌린 돈을 자기자본으로 오인). 대출금을 현금으로 들고 있으면 증자로 따로 기록. **남은 것:** 상환/이자 이벤트로 잔액 자동 감소, 순자산 추이(자산−부채 시계열), 대출로 산 자산 추적(레버리지 매입 플래그), 외화 부채. 관련: [[xirr-founding-model]](XIRR은 투자계좌만 — 부채는 순자산 표시값이지 XIRR 현금흐름 아님), spec xirr §범위.

  - **DART 공시 피드 (구현됨 v1):** `lib/finance/dart.ts` — OpenDART(무료, `.env.local` `OPENDART_API_KEY`). 한국 6자리 종목만. **stock_code→corp_code 매핑**: corpCode.xml(zip)을 node:zlib `inflateRawSync`로 해제(의존성 0) 후 **`<list>` 블록 단위 파싱**(corp_name↔stock_code 사이 corp_eng_name 끼어 있음 주의) → 모듈 메모이즈. `getDisclosures`(종목별)·`getDisclosuresForSymbols`(집계). 노이즈 공시(특정증권등소유/최대주주변동) 제외, 6h 캐시. **표시:** CFO 리포트 "이번 분기 주요 공시" + 종목 상세 "최근 공시"(제목+날짜+DART 원문 링크). **해석 힌트(펀더멘털 플래그 §11, 규칙 기반·토큰 0, 구현됨):** `disclosureHint(title)` — 공시 제목 키워드→"의미 힌트"(유상증자→지분 희석, 합병→구조 변경, 소송→우발부채, 자사주→주주환원 등) 질문형(단정 X) + 톤(warn 하락색/good 상승색/info). 💡로 공시 아래 표시. 인터페이스 seam(토스 교체 가능). **공시 전체보기(구현됨):** `getDisclosurePage`(pblntf_ty 유형필터 + 페이지네이션 + total) → `/stocks/[symbol]/disclosures`(유형 칩 전체/정기A/주요사항B/발행C/**지분·대주주D**/거래소I/기타E + 이전/다음, 최근 3년). 정기(A)=사업·분기보고서, **지분·대주주(D)=임원·최대주주 지분 매매·5% 대량보유**(대주주 매도 체크 — 힌트 부착). 종목 상세 "최근 공시 → 전체 보기" 진입. 공유 `DisclosureList` 컴포넌트. **남은 것:** **AI 요약 "핵심 3줄"(V3, CFO 유료층)**, 알림 배너·푸시("내 사업부 소식 ●N"), 보유종목 통합 IR 피드.
  - **종목별 펀더멘털 (구현됨 v2 — account_id 정확도 보강):** `getFundamentals(stockCode, year)`(dart.ts) — **`fnlttSinglAcntAll.json`(전체계정) + account_id(IFRS/DART 표준코드) 결정적 추출**(한글이름 매칭보다 신뢰도↑). 주의: 영업이익=`dart_OperatingIncomeLoss`(DART 확장태그), 순이익·자본=`...AttributableToOwnersOfParent`(지배주주 우선). 현금흐름표 포함 → **FCF=영업활동현금흐름−총CapEx**(유형+무형 취득). ⚠️ **오너이익(§12-1)=순이익+감가상각(D&A)−총CapEx ≠ FCF** — D&A가 한국 DART 본문엔 없음(6/6 검증: 대기업은 `AdjustmentsForReconcileProfitLoss(조정)` 한 줄로 합산, 주석에만) → **D&A 수기 입력 브릿지 (구현됨)**: `manual_fundamentals` 테이블(holding·symbol·dna·year, RLS, upsert) + `DnaInput` 컴포넌트(종목상세) + `setManualDna`/`clearManualDna`. 순이익·CapEx는 DART 자동, **D&A 한 칸만 수기**(주석에서) → 오너이익 산출. 미입력이면 FCF 폴백 + "+ 감가상각 입력해 오너이익 보기". 미국 EDGAR는 D&A 자동(다음). 재료(순이익·OCF·총CapEx·D&A)는 정직 표시(§4·§8). 포지셔닝: 옵트인 파워 기능(점진적 공개) — 가치투자자·창업자 도그푸딩·CFO 유료층. 코어엔 안 얹음. **다음:** §12 내재가치·안전마진(=오너이익/(10년물×2, floor 8%)) + RONTE(다년). + `stockTotqySttus`(유통주식수)→EPS·PER. **금융주 감지**(ifrs-full_Revenue 없음→isFinancial, UI 주의문구·부채비율 해석 경고). **검증**(자산≈부채+자본 ±1% + 핵심항목 존재 → confidence high/low, low면 ⚠). **종목 상세** 핵심비율4종 + **오너이익 박스** + 규모 + 금융주/신뢰도 문구. 검증: 삼성 오너이익 21.6조·ROE(지배)8.6%, KB금융 isFinancial✓·부채비율1206%(플래그). 연결CFS 우선·OFS 폴백·최신연도 역순. **남은 것(다음):** **미국 SEC EDGAR**(us-gaap concept, ticker→CIK `company_tickers.json`, NetIncomeLoss=지배주주·CapEx/D&A 있음, 무료·키X — 6자리아니면 EDGAR로 분기) / **투시 합산(Level B)** 지분율 가중 ROE·투시매출 "내 사업의 실체"(CFO 유료층) / PBR·성장률 / 섹터태그 / 정밀화시 FnGuide 라이선스(seam).
    - **펀더멘털 플래그 §11 (구현됨 v1 — "재무제표가 던지는 질문"):** `lib/finance/fundamentalFlags.ts` `computeFundamentalFlags(series)` 순수함수(규칙 기반·토큰 0). **1급(이익의 질):** F1 순이익↑·영업현금흐름↓ / F2 순이익 ≫ 영업이익(일회성·영업외) / F3 매출채권이 매출보다 급증(밀어내기) / F4 재고가 매출보다 급증(떨이 임박). **2급:** F5 이자보상배율<1(좀비). 최신·직전 연도 YoY 비교(F2·F5는 단년). **데이터는 기존 펀더멘털 재사용** — `Fundamentals`에 `receivables`(매출채권)·`inventory`(재고) 추가(DART byNmBS·EDGAR AccountsReceivableNetCurrent/InventoryNet). **★ 단정 금지** — 항상 "신호 + 질문 + 확인처"(같은 신호도 좋고 나쁨이 갈림). 금융업(isFinancial)은 해석이 어긋나 엔진이 건너뜀. **표시 = 기존 "재무 건강 체크"와 통합**(`FinancialHealth` 컴포넌트, 별도 섹션 아님): 종목 상세 "**재무제표 자세히**" 펼침 **최상단**에 [재무 건강 체크 5/5 체크리스트(흑자·이익의 질·이자체력·성장·현금흐름, 상태 pass/fail)] + 그 아래 [⚠️ 확인할 거리=플래그 목록]을 한 블록으로. 플래그 없으면 체크리스트만(5/5가 곧 안심). **재무제표 순서 변경: 재무상태표 → 손익계산서 → 현금흐름표**(자세히 펼침 기준). 건강체크=현재 상태, 플래그=YoY 변화·괴리로 역할 분리(이자보상배율 중복은 체크=≥5배/플래그=<1배로 강도 차이). 테스트 `fundamentalFlags.test.ts` 10케이스. **CFO 리포트 문장 연결은 보류(CFO 보류 결정).** **남은 것:** 투시 상세 전체요약, 매출채권/재고 회전일수 정밀화.
    - **ETF 메타(총보수·추종지수) — (v1 부분 구현):** ETF **총보수(TER)** = 가치투자=비용 최소화 직결(수수료 랭킹의 ETF판). 야후 미제공(quoteSummary crumb 필요·불안정, 확인됨) → **카탈로그 상수**(`CatalogItem.ter`, 참고용 공시값 — 운용사 인하 잦아 정확성 미보장, UI "참고" 표기). 표시: 검색/추천 행("총보수 X%") + ETF 상세("총보수(연, 참고)"). 같은 지수 내 **저보수 상품 위주로 추천 선별**. **남은 것:** 정확값 자동화(KRW/운용사 소스), 추종지수·AUM·괴리율, ETF 투시(구성종목 룩스루)와 묶음.
  - ⚠️ **원칙:** 이 제품은 풀 자산관리 엑셀의 1:1 복제가 아니다. 브랜드를 강화하는 항목만 선별 — "한 화면 한 가지 일 + 게임화" 정체성 유지.

각 STEP은 동일하게 "그 STEP만, 끝나면 멈추고 요약" 원칙으로 진행한다.

## 부록 — 출시 전 필수 체크리스트 (MVP 솔로 테스트엔 불필요, 정식 공개 전 필수)

아래는 너 혼자 테스트하는 MVP 단계에선 없어도 되지만, **일반 사용자에게 공개(베타→정식)하기 전에는 반드시 갖춰야 하는 것들**이다. 데이터·화면이 깨지는 항목이 아니라 법적·계정 관리 항목이라 V0 구현에서 분리한다.

- **이용약관·개인정보처리방침 동의** — 법적 필수. 회원가입 플로우에 동의 단계 추가. (한국 개인정보보호법: 수집·이용 동의 필수)
- **비밀번호 재설정 / 로그인 복구** — 매직링크만 쓰면 일부 자동 해결되나, 계정 접근 실패 경로를 점검.
- **계정 삭제 / 데이터 삭제 요청** — 사용자 권리(삭제권). 설정에 추가.
- **가입 중 이탈 재개** — 모드는 골랐으나 설립 미완료한 사용자를 멈춘 단계부터 재개(레일 0-1, V1).
- **문의·신고 채널** — 최소한의 이메일 연락처.

이 체크리스트는 V0 STEP 0~9에 포함하지 않는다. 정식 공개 직전 별도 단계로 처리.

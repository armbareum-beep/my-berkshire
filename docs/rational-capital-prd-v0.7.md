## 0. 이 MVP가 검증하는 단 하나의 가설

> **진지한 투자자는 정확한 수익률과 "투자회사 CEO 정체성"을 얻기 위해, 투자 행동(매수·매도·배당·입출금)이 일어난 시점에 바로 기록할 것인가.**

이 문장이 모든 의사결정의 기준이다. 어떤 기능이든 이 가설 검증에 기여하지 않으면 v0에서 제외한다.

**측정 지표:** 가입 후 30일 내 투자 이벤트를 3회 이상 기록한 사용자 비율.

## 1. 문제 정의

개인 투자자는 자신의 실제 성과를 정확히 알 수 없다.

**① 계좌 중심 관점** — 사람들은 증권 계좌, ISA, 연금, 현금을 따로 본다. 정작 관리해야 하는 건 **"나의 자본"** 전체다.

**② 수익률 왜곡** — 대부분의 앱은 평가손익만 보여준다. 진짜 성과는 입금·출금·배당·매수·매도의 *시점*을 반영한 XIRR이라야 한다.

**③ 투자 기록의 부재** — 사람들은 언제, 왜, 무엇을, 얼마에 샀는지 시간이 지나면 기억하지 못한다.

---

## 2. 비전

개인 투자자를 투자회사의 CEO로 만든다. 사용자는 투자자가 아니라 자신의 자본을 운영하는 **Capital Allocator**다. 우리가 파는 건 자산조회 앱이 아니라 **개인 투자회사 운영 시스템**이다.

**제품 목표 — 수익보다 영향력.** 이 제품의 1차 목표는 매출이 아니라 **창업자(이바름/rational.cap)의 투자 철학을 널리 퍼뜨리는 것**이다. 앱은 수익 사업이기 이전에, "가치투자는 실력이다 · 주가가 아니라 사업을 보라 · 마찰비용이 복리를 죽인다 · 투자엔 여러 길이 있다"는 세계관을 사람들이 *직접 체험*하게 만드는 매개다. 따라서 모든 결정의 나침반은 **"이것이 영향력(확산)에 도움이 되는가?"**이다.

이 목표에서 전략이 따라온다:
- **무료 최대 확산 우선.** 과금은 성장을 막으므로 후순위(운영비 감당이 어려워질 때만). 사용자를 *지불자*가 아니라 *전파자*로 본다.
- **바이럴·공유가 핵심 동력.** 연차보고서·백분위 카드·스타일 칭호·CFO 리포트는 "자랑하고 퍼뜨리는" 장치이자 철학 전파 수단.
- **rational.cap ↔ 앱 선순환.** 앱이 콘텐츠 소재를 생산하고, 콘텐츠가 앱 유입을 만든다. 창업자의 Threads 영향력과 앱이 한 몸.
- **수익은 영향력 확보 후 자연히 따라온다.** 구독·계좌 리워드·자문은 그다음(13번). 토스가 "송금 무료"로 영향력부터 키운 경로와 같다.

각 핵심 기능은 곧 철학의 구현이다: 투시 재무제표=「주가 말고 사업을 보라」, 마찰비용=「과매매는 독이다」, CFO 리포트=「철학을 가진 참모」, 스타일 칭호=「투자엔 여러 길이 있다」.

## 3. 핵심 컨셉

**나만의 지주회사** — 가입과 동시에 자신만의 투자회사를 설립한다. (예: 바름캐피탈, ENUF 홀딩스, Arcana Holdings)

**나는 투자자가 아니다** — 나는 자본배분가, CEO, 투자회사 대표다.

**투자 행동이 곧 회사 역사다** — 모든 행동이 회사 연혁으로 기록된다. (2026.03 삼성전자 첫 투자 → 2026.06 자산 1천만원 돌파 → 2026.09 KOSPI 초과수익)

컨셉의 영혼은 규모가 아니라 주당 복리수익률이다. "나는 좋은 자본배분가인가, 아니면 그냥 많이 저축했나"에 답하는 게 XIRR이고, 그게 이 제품의 본질이다.

## 4. 입력 모델 — 단일 이벤트 드리븐

**모드는 하나다. 입력은 다섯 가지 이벤트뿐이다.**

| 이벤트 | 게임 명칭 | 필드 |
|---|---|---|
| 매수 | 자회사 인수 (M&A) | 종목 · 수량 · 가격 · 날짜 |
| 매도 | 지분 매각 | 종목 · 수량 · 가격 · 날짜 |
| 배당 | 배당금 수령 | 종목 · 금액 · 날짜 |
| 입금 | 증자 | 금액 · 날짜 |
| 출금 | 자본 인출 | 금액 · 날짜 |

투자 행동이 일어난 시점에 바로 기록한다. "삼성전자 100주 인수 체결" 같은 즉각적 피드백이 입력의 보상이다.

이 다섯 이벤트 + 시세 API만으로 모든 대시보드 숫자가 자동 계산된다. 사용자가 평가액·수익률·현금 잔고를 직접 입력하는 필드는 존재하지 않는다.

> **v0.4의 Quick Mode(총 평가액 스냅샷 입력)는 폐지한다.** 정확성이 핵심 무기인 제품이 "총액만 넣는 부정확한 모드"를 함께 제공하면 메시지가 흐려진다. 정확하거나, 없거나 둘 중 하나다.

### 유일한 예외 — 회사 설립 등기 (온보딩 1회)

신규 사용자는 과거 거래 이력을 전부 입력할 수 없다. 가입 시 한 번만:

- 투자 시작일 (설립일)
- **현재 보유 종목** — 각 종목의 **수량 + 평단(평균 매입가) [필수]** + **매수일 [선택·권장]**. ← 투입 원금·현금흐름이 여기서 나온다.
- **(선택) 현금 보유분** — 아직 주식을 안 산 여윳돈(예수금). 없으면 0.

**"초기 자본금"은 따로 묻지 않는다(중복).** XIRR에 필요한 건 "현금흐름의 날짜+금액"이고, 그건 **각 매수(평단×수량+매수일)가 곧 그날 투입한 원금**이므로 보유 종목 입력만으로 원금이 자동 계산된다. 별도 "자본금" 숫자를 또 받으면 돈을 두 번 세는 셈. 주식 외 현금은 "현금 보유"로 분리해 받는다(자본금이 아니라 현금 잔고라는 다른 개념).

**보유 종목 추가 = 거래 입력과 동일한 검색 시퀀스 재사용:** 설립의 종목 추가는 별도 UI가 아니라 **거래 입력 컴포넌트(검색 → 평단 → 수량 → 매수일)를 그대로 호출**한다. 한 번 만든 입력 흐름을 온보딩에서 재사용(코드·UX 일관성).

**수익 계산 — 평단 필수, 날짜 선택(하이브리드):**
- **평단(평균 매입가)은 필수.** 평단이 곧 "어디서(얼마에) 샀는지"이므로, 현재가−평단으로 손익이 계산된다. 평단 없이 수량만 받으면 평가액만 보이고 수익률이 안 나온다.
- **매수일은 선택(권장).** XIRR엔 날짜가 필요하지만 옛 매수일을 모르는 사용자가 많고 강제하면 이탈한다. 알면 입력→정확한 XIRR, 모르면 비움→평단을 설립일 기준점으로 XIRR 근사.
- **분할 매수(lot) — 간편 기본 + 정확 선택(하이브리드):** 같은 종목도 여러 번 나눠 사면 매수일·단가가 건별로 다르다(100주 = 1월 30주 + 3월 20주 + 6월 50주). 정확한 XIRR엔 건별 입력이 맞지만 번거롭다. 그래서: **기본 = 평단 하나 + 대표 날짜 하나(간편, 근사)**. 더 정확히 원하면 **"여러 번 나눠 샀어요" → 매수 건(lot)을 날짜·수량·단가별로 추가**(정확). 귀찮으면 평단으로 끝, 정확을 원하면 쪼갠다. 이 분할 입력의 궁극 해법은 V3 증권사 연동(매수 건별 자동 수신)이며, 수동 단계(V0~V1)에선 이 하이브리드로 충분.
- **모드별:** 챌린지=매수일 오늘 자동(평단=현재가). 장부=하이브리드. 라이브(V3)=증권사 연동으로 평단·날짜·lot 자동.

이것은 반복되는 스냅샷이 아니라 **1회성 개업 등기**다. 이후의 모든 기록은 이벤트 드리븐으로만 쌓인다. 설립 등기 이전 기간의 수익률은 계산하지 않는다(데이터가 없으므로 — 정직하게 "설립 이후 성과"만 보여준다).

> 주의: 설립 등기 = "보유 종목을 평단과 함께 등록"이지 "총 평가액만 입력"이 아니다(v0.4 Quick Mode 폐지 정신). 평단 없이 수량만 받으면 평가액만 보이고 수익률이 안 나오므로 평단은 반드시 받는다.

### 이벤트 수정·삭제 (모드별 — 무결성의 핵심)

잘못 입력한 거래를 고칠 수 없으면 사용자가 갇힌다. 단, 자유로운 과거 수정은 챌린지·라이브의 소급 조작 차단을 무너뜨린다. 그래서 모드별로 다르다:

- **장부(ledger):** 이벤트 자유 수정·삭제 허용(자기 기록이고 남과 안 겨루므로 안전).
- **챌린지·라이브(challenge·live):** **입력 당일은 자유 수정·삭제 허용**(오늘 친 오타를 오늘 고치는 건 시간 위조가 아니다 — 여전히 오늘). **날짜가 지난 과거 기록은 상쇄(reversing entry)만 허용** — 원본은 남기고 "취소" 이벤트를 오늘 날짜로 추가해 효과를 0으로. 과거 이벤트의 직접 수정·삭제는 금지(소급 조작 차단). 명백한 오타는 "취소 후 재입력"으로.
- 어느 모드든 **삭제/취소 이력은 보존**(감사 추적). 회사 연혁에는 정상 기록만 노출하되 데이터는 남긴다.

## 5. 핵심 데이터 모델

모든 숫자는 아래 데이터에서만 계산된다.

### Holding (투자회사)
- 회사명 / 설립일 / **설립 등기 데이터(초기 자본금 · 설립 시점 평가액)**
- **`mode` (운영 모드 — 신설):** `ledger`(장부) / `challenge`(챌린지) / `live`(라이브). 데이터의 검증 수준을 결정하며, 이에 따라 경쟁 범위(자기 vs 남)가 갈린다(5-2 참조). v0은 ledger·challenge 구현, live(라이브)는 V3.

### Account (계좌)
- 이름 / 소유자 구분(본인 · 배우자 · 자녀)
- **`account_type` (과세 유형 — 세금 규칙이 여기 종속):** `GENERAL`(일반) · `ISA` · `PENSION`(연금·IRP) · `OVERSEAS`(해외). 3단계 참조.
- **`commission_rate` (수수료율 — 계좌 개별 설정):** 증권사·상품마다 다르므로 사용자가 계좌 생성 시 한 번 등록(예: 0.015%). 이후 그 계좌의 모든 매수·매도에 자동 적용되어 `fee_and_tax`로 흘러든다.
- 소속 Holding (FK → holdings.id)
- v0에서는 가입 시 기본 계좌 1개('메인', GENERAL)가 자동 생성된다. V1 멀티계좌 확장을 위한 사전 설계.

> **수수료 vs 세금 분리 원칙:** **수수료율 = 계좌 개별 설정(증권사마다 다름)**, **세금 규칙 = 계좌 유형 종속(국가가 정함, 증권사 무관)**. 둘은 서로 다른 출처를 가지며, 합쳐져 Event의 `fee_and_tax`가 된다.

### Event (투자 이벤트) — *CashFlow + Transaction 통합*
- 유형: 매수 / 매도 / 배당 / 입금 / 출금
- 종목(매수·매도·배당만) / 수량(매수·매도만) / 가격 또는 금액 / **날짜 필수** / 소속 Account (FK)
- **종목은 티커(symbol)로 저장한다.** 로고·종목명·현재가는 저장하지 않고 티커로 시세 API(토스 종목 마스터/로고)에서 조회한다. UI는 종목이 나오는 모든 곳에서 토스증권식 행(원형 로고 + 종목명 + 티커 + 등락)으로 표시(디자인 전략 4-1).
- **`fee_and_tax`(수수료·세금, 기본 0):** 해당 거래에서 빠져나간 수수료와 세금의 합. 사용자가 비워두면 시스템이 계좌 수수료율 + 계좌 유형별 세율로 자동 추정해 채운다(2·3단계 참조). 이 값은 현금 잔고를 깎는다.
- → 매수·매도는 보유 포지션을, 입금·출금·배당은 XIRR 현금흐름을 만든다

### 순자산 vs 투자수익률 — 두 숫자를 분리한다 (뱅크샐러드형 자산 분류 + 부채)

뱅크샐러드처럼 "순자산 = 자산 − 부채"를 정직하게 보여주되, **"내 전체 재산"과 "내 투자 실력"은 다른 질문**이므로 두 숫자를 섞지 않는다. 진짜 지주회사는 차입(레버리지)이 자본구조의 핵심이므로 부채 없이는 순자산이 거짓말이 된다.

- **자산 분류:** 투자(증권 계좌) · 현금 · 연금 · 부동산. account_type에 이미 깔린 분류(GENERAL/ISA/PENSION/OVERSEAS + 향후 REAL_ESTATE)를 그대로 활용. 대시보드 자산배분을 이 분류로 표시.
- **부채(신설 개념):** 대출 등 음(−)의 자산. 순자산에서 차감.
- **순자산 표시** = 투자 + 현금 + 연금 + 부동산 − 부채 (전체, 뱅샐형).
- **XIRR(투자 실력)** = **투자 계좌만**. 부동산·부채·연금은 XIRR에서 제외 — 부동산 18억이 끼면 수익률이 부동산에 압도당해 "내 투자 실력"이 안 보인다. 부동산은 별도 평가 파이프라인(국토부 실거래가, §로드맵 V2), XIRR 정밀도를 오염시키지 않는다.
- **차별점:** 뱅샐은 "전체 재산"만 보여주고, 우리는 *전체 재산(순자산)과 투자 실력(XIRR)을 둘 다, 단 안 섞어서* 보여준다.
- **범위:** v0은 투자 중심(투자·현금). **부채·부동산·자산 분류 통합 표시는 V1**(부동산은 V2 부동산 사업부와 연계). v0 데이터 모델엔 부채를 담을 수 있게 자리만 설계.

### Position (보유 포지션) — *Event에서 파생, 캐시 성격*
- 종목 / 보유수량 / 평균단가 / 소속 Account
- Event 누적으로 자동 계산. 사용자가 직접 수정 불가.

> **불변 원칙:** 대시보드의 모든 숫자는 Event(+ 설립 등기)에서만 파생된다. 현재 평가액 = Σ(보유수량 × 실시간 시세) + 현금 잔고. 현금 잔고 = 입금 − 출금 + 배당 + 매도대금 − 매수대금 **− Σ(수수료·세금)** (이벤트에서 자동 산출). 비용은 현금 잔고를 통해 평가액에 반영되므로 XIRR 수식 자체는 바뀌지 않는다 — 배당과 동일한 경로다.

## 5-1. 비용·세금 규칙 — 계좌 유형 종속

`fee_and_tax`는 두 출처가 합쳐진 값이다.

**① 수수료 (계좌 개별):** Event 거래대금 × Account의 `commission_rate`. 매수·매도에만 적용.

**② 세금 (계좌 유형별):**

| account_type | 매도 시 | 배당 시 | 비고 |
|---|---|---|---|
| **GENERAL** (일반) | 증권거래세 | 배당소득세 15.4% | 기본형 |
| **ISA** | 손익통산 후 한도까지 비과세, 초과분 분리과세 | 통산 대상 | 만기 전 과세 이연 |
| **PENSION** (연금·IRP) | 운용 중 비과세 | 운용 중 비과세 | 과세 이연 + **납입액 세액공제(마이너스 비용)** 트래킹 |
| **OVERSEAS** (해외) | 양도세 22% (연 250만원 공제) | 현지 원천징수 후 국내 정산 | |

> **세율은 하드코딩 금지.** 증권거래세 등은 해마다 바뀌므로 **전역 설정값(config)** 으로 빼서 한 줄로 갱신 가능하게 한다. (세율 자체는 증권사 무관 — 국가가 정함.)

### 제품 관점 — 이건 부담이 아니라 무기다

연금계좌가 세액공제를 받으면, 그건 **"정부 보조금을 받는 자회사 사업부"** 다. 일반계좌보다 세후 수익률이 구조적으로 높게 나온다. 사용자가 이걸 숫자로 보는 순간, 제품이 수익률 계산기를 넘어 **자본배분 의사결정 도구**가 된다. 세제 혜택 계좌를 전략적 자회사로 보게 만드는 것 — 이게 "나만의 버크셔" 컨셉의 확장이다.

### MVP 범위 컷

`account_type` 필드는 v0부터 만들되, **세금 로직은 GENERAL만 구현한다.** ISA 손익통산·연금 인출 과세는 계산이 복잡하므로 V1 멀티계좌 단계로 미룬다.

- **v0:** GENERAL 세금만 — 배당세 15.4% + 거래세. 수수료 자동 적용.
- **V1:** ISA·PENSION·OVERSEAS 유형별 세금 규칙 + 세액공제 트래킹. (Account 레이어가 이미 있으니 확장만)

## 5-2. 운영 모드와 리그 — 검증 수준에 따라 경쟁 범위가 갈린다

게이밍에 순위가 붙는 순간, **자기 신고 데이터는 조작 가능**해진다(과거 날짜로 "그때 샀음" 위조). 그래서 데이터의 *검증 수준*에 따라 누구와 경쟁할 수 있는지를 분리한다. 검증될수록 남과 겨루고, 안 될수록 자기와 겨룬다 — 이 비례가 신뢰의 근거다.

핵심 구분: **시간 검증(오늘 고정, 소급 불가)과 존재 검증(증권사가 거래의 실재를 보증)은 다르다.** 오늘 고정은 날짜 조작만 막고, 거래가 진짜인지는 못 막는다. 리더보드(실전)엔 존재 검증이 필요하다.

| 모드 | 데이터 | 검증 | 거래일 | 경쟁 범위 | 시점 |
|---|---|---|---|---|---|
| **ledger (장부)** | 과거 포함 자기 기록 | 미검증 | 날짜 선택 가능 | **자기 자신·시장**(과거의 나, KOSPI, 버크셔). 남과 섞지 않음 | V0 |
| **challenge (챌린지)** | 오늘부터 가상 기록 | 시간 검증(모두 가상이라 공평) | **오늘 고정(소급 불가)** | **남과 경쟁 가능**(가상끼리 공평) | V0 |
| **live (라이브)** | 연동된 본인 계좌 | 존재 검증(증권사 보증) | 연동 데이터(자동) | **남과 경쟁 가능**(진짜 순위) | V3 |

**왜 이렇게 갈리나**
- **장부가 남과 못 겨루는 이유:** 오늘 고정이어도 "아내 계좌로 오늘 100주 샀다"가 진짜인지 시스템이 모른다. 존재 검증이 없으므로 리더보드 부적격. 대신 *자기 자본 운영·자기 과거와의 비교*에는 완벽히 유효하다.
- **챌린지가 남과 겨룰 수 있는 이유:** 전원이 가상이고 전원이 오늘 고정이라 **공평**하다. 돈이 안 걸려 조작 동기도 약하고, 가상이라 존재 검증이 불필요하다.
- **실전만 진짜 순위인 이유:** 증권사 연동이 거래의 실재를 보증한다. 가족 계좌는 평생 본인 연동이 안 되므로, 가족이 순위를 원하면 각자 연동(V2 Family Holding).

**전략적 가치**
- **챌린지 = 진입장벽 0.** 계좌·거래내역·돈 없이 "오늘부터 가상 투자"로 시작. 돈 없는 사람·학생·투자 입문자까지 첫 유저 풀이 폭발. rational.cap 철학("실력이지 돈 크기가 아니다")의 증명이자 콘텐츠 소재.
- **온보딩 퍼널:** 챌린지로 시작 → 실력 검증 → 라이브 계좌(투권인 계좌개설 리워드) → 연동. 모드가 곧 성장 깔때기.
- **소급 불가라 챌린지도 진짜다:** 일반 모의투자 앱은 "과거에 샀으면 떡상" 자랑이 무의미하지만, 우리는 오늘부터만 기록되므로 챌린지 수익률도 실력의 증명이 된다.

**범위 원칙(중요): 데이터 구조는 지금, 화면은 나중.**
- v0: `mode` 필드(ledger/challenge) + "챌린지·라이브는 거래일=오늘 고정" 규칙을 **데이터·로직에 심는다.** XIRR 엔진은 모드 무관 공유.
- **리그·순위 UI는 명세로만 두고 코드는 V1**(유저 간 순위는 사람이 모여야 의미). 지금 순위 화면을 만들면 텅 비어 역효과. (account 레이어를 미리 깔아둔 것과 같은 "구조 먼저, 화면 나중" 전략)
- 챌린지의 *시장·과거의 나* 비교는 유저 0명이어도 작동하므로 v0부터 가능. *유저 간* 순위만 V1.

**모드별 랭킹 페이지 (V1) — 모드는 섞지 않는다.**
랭킹은 모드별로 *분리된 페이지*로 운영한다. 검증 수준이 다른 데이터를 한 리더보드에 섞으면 불공정하기 때문(섹션 도입부의 핵심 원칙).
- **챌린지 랭킹:** 챌린지 모드 유저끼리만. 전원 가상·오늘 고정이라 공평. 🟢 챌린지 배지. v0 이후 유저 모이면 가장 먼저 켜지는 랭킹.
- **라이브 랭킹(V3):** 라이브 모드(연동·존재 검증)끼리만. 🔵 라이브 배지. 진짜 돈 진짜 순위.
- **장부:** 랭킹 페이지 없음. 미검증·소급 가능이라 남과 겨루지 않는다. ⚪ 장부는 "자기 자신·시장(과거의 나·KOSPI·버크셔)" 비교만, 리더보드 부적격.
- 각 랭킹 페이지: 내 순위(백분위) + 상위권 + 검증 배지 표시. **유저 수 임계치(수천 명) 도달 전엔 비활성**(통계적으로 무의미). 진입은 리그/순위 화면에서 모드 탭으로 전환.

## 6. 시세 피드 — 토스증권 Open API

- **역할:** 보유 포지션 × 실시간 시세 = 현재 평가액 자동 계산. 사용자가 "지금 얼마야"를 입력할 필요가 없게 만드는 핵심 의존성.
- **커버리지:** 국내(KRX) + 미국 주식, REST + WebSocket, OAuth 2.0 Client Credentials.
- **상태:** 2026년 6월 현재 사전 신청 단계적 롤아웃 중. **즉시 사전 신청하고, 개발 초기엔 목업 시세로 진행.** 승인 지연 시 폴백: 코스콤 오픈API 또는 한국투자증권 REST API.
- **시세 실패 시 화면 상태(필수):** 시세를 못 받으면(설립 직후, API 미승인, 네트워크 오류) **0이나 이전 값으로 조용히 대체하지 않는다**(XIRR 명세 엣지케이스 3과 일치). 대신 평가액·수익률 자리에 "시세 불러오는 중" 스켈레톤 또는 "시세 갱신 필요" 상태를 명시적으로 표시하고, 재시도 버튼을 둔다. 첫 화면이 깨져 보이지 않게 하는 것이 목적. 목업 시세로 개발하는 동안에도 이 상태 분기는 구현해 둔다.

## 8. 동일 현금흐름 벤치마크 (PME)

`내 수익률 − KOSPI 수익률`은 틀린 계산이다. ENUF는 **동일한 날짜에 동일한 금액을 벤치마크에 넣었다면 지금 얼마였을지**를 계산해 비교한다. "내가 타이밍이 아니라 실력으로 시장을 이겼는가"를 정밀하게 측정한다.

**대시보드 표시(신설):** XIRR 카드 아래 벤치마크 비교 한 줄을 둔다 — 예: "S&P500 +9.0% / 나 +18.4% — 9.4%p 앞섰습니다." 숫자가 그냥 18.4%가 아니라 *시장을 이긴* 18.4%가 되게 만드는 핵심 장치.
- **기본 벤치마크 = S&P500.** 투자 철학(버핏·버크셔·가치투자)이 미국 시장 기준이고, S&P500이 "이기기 어려운 진짜 기준"의 대명사이기 때문.
- **토글로 코스피·버크셔 전환.** 셋 다 같은 PME 엔진 재활용(추가 비용 거의 0). "버핏(버크셔)도 이겼나?"가 가장 강력한 자랑거리.
- **벤치마크 = 2그룹 (벤치마크 상세 화면에서 전체 비교):**
  - **지수 그룹(실시간, V0.5):** S&P500(기본)·나스닥·코스피 등. 공개·실시간 데이터, PME 엔진 직결. "시장 평균을 이겼나."
  - **레전드 그룹(큐레이션, 공개 기준, V1):** 버크셔(버핏)·국민연금 등 *내가 존경하는 거장들만* 큐레이션. "거장을 이겼나" — 가장 강력한 자랑거리이자 §9 분석카드("버핏을 이겼습니다")와 결합. 주의: 펀드 수익률은 분기/월 단위로 늦게 공개되므로 "최근 공개 기준" 시점을 정직하게 표기(예: "버크셔 2025 4Q 기준"). 아무 펀드나 넣지 말고 큐레이션이 곧 브랜드.
    - **추가 방식:** 우리가 *공개 수익률 데이터가 있는* 거장/기관 풀을 기본 제공(버핏·멍거 등 영웅은 기본 탑재). 사용자는 그 풀에서 라이벌을 골라 추가(ARK·타이거펀드 등). **사용자 임의 수익률 직접 입력은 금지** — 공개·검증된 데이터가 있는 대상만(조작 가능한 죽은 숫자 방지, 데이터 무결성 원칙 일관). 풀 관리는 우리 몫이고 큐레이션이 신뢰를 만든다.
  - 대시보드엔 기본 1개(S&P500)만, 전체 목록·비교는 벤치마크 상세 화면에서.
- **비교는 수익률(%) 대 수익률(%).** 내 XIRR(%)와 벤치마크 수익률(%)을 직접 비교하므로 **통화 환산이 필요 없다**(퍼센트는 단위 없는 비율). 환율 환산은 절대 금액 비교에만 필요하고, 우리는 비율로만 비교하므로 무시한다. (엄밀히는 달러 지수 상승 기간의 원화가치 변동 효과가 있으나 둘째 자리 정밀도 문제로 MVP에서 무시.)

## 8-1. 마찰비용 (Cost of Trading) — 숨은 비용을 정면에 띄운다

수수료·세금은 현금 잔고에 녹아 XIRR에 **이미 정확히 반영**된다(수익률은 틀리지 않는다). 문제는 그 비용이 *보이지 않는다*는 것. "나만의 버크셔" 컨셉에서 과도한 매매는 죄악이다 — 버핏·멍거의 핵심 교리가 "마찰비용이 복리를 죽인다"이므로, 우리는 이 비용을 숨기지 않고 **정면에 띄워 사용자를 각성시킨다.** 이것이 단순 계산기와 "좋은 투자 습관을 만드는 도구"의 차이다.

**마찰비용의 3요소 (마찰비용 카드를 누르면 상세 화면에서 종목별로 펼쳐 본다):**
- **매매 수수료** — 사고팔며 낸 합계(fee_and_tax 중 수수료분).
- **세금** — 배당세·거래세 합계.
- **ETF 총보수(TER) — 신설.** ETF가 *보유하는 동안* 매년 까먹는 운용보수(예: KODEX 200 연 0.15%). 매매 수수료와 성격이 다르다(거래가 아니라 시간이 녹이는 비용 — 가장 교활한 마찰). 티커로 종목 마스터에서 보수율 조회(사용자 입력 없음). 보유 ETF별로 "연 보수율 · 올해 ₩X · 10년 누적 ₩Y" 환산 표시. **명목 보수율 ≠ 실질부담비용**(더리치 사례: 보수 0.01%인데 실질부담 0.1017%)이므로 가능하면 실질부담비용까지, 같은 추종 ETF끼리 "어디가 진짜 싼가" 순위 제공(V1~V2).
- 맨 아래 **"수수료 드래그"** 한 줄: 이 비용들이 수익률을 매년 몇 %p 깎는지. → 과매매 경고(CFO 리포트 멍거 톤)와 연결.

데이터: ETF 보수는 티커로 조회(공짜). 범위: 마찰비용 인사이트는 fast-follow(V0.5~V1).

**지표 (전부 Event의 `fee_and_tax`와 매매 횟수에서 파생 — 새 데이터 불필요):**

- **누적 마찰비용** — "설립 이후 수수료·세금 총 ₩320,000"
- **회전율(turnover)** — 평균 자산 대비 매매 규모. 얼마나 자주 사고팔았나.
- **수수료 드래그** — "잦은 매매로 연 수익률이 약 1.2%p 깎였습니다." *(계산: 비용을 0으로 가정한 가상 XIRR − 실제 XIRR. 같은 XIRR 엔진 재사용.)*

가장 강력한 건 드래그다. 추상적 비용을 "네 수익률에서 깎인 %p"로 환산해 보여주면 행동이 바뀐다.

## 8-2. 투시 재무제표 (Look-Through Financials) — 내 지주회사의 진짜 실적 ★

별도 명세: `lookthrough-financials-spec-v1.md`. 주가가 아니라 **내가 소유한 사업체들의 실제 펀더멘털**로 지주회사를 평가한다. 버핏의 look-through earnings 개념의 전면 구현.

- **원리:** 지분율 = 보유수량 / 총발행주식수. 이를 각 회사 재무항목에 곱해 "내 몫"을 구하고 보유 종목 전체를 합산.
- **투시 손익(P&L):** 매출 · 영업이익 · 순이익(`Σ EPS × 보유수량`).
- **투시 재무상태표(B/S):** 자산 · 부채 · 자본.
- **투시 현금흐름표(C/F):** 영업·투자·재무 활동현금흐름 + **잉여현금흐름(FCF = 영업현금흐름 − CapEx)**. 현금은 분식이 어려워 가치투자자가 가장 신뢰하는 표. **FCF는 추정 없이 공시값으로 계산**되므로(owner earnings와 달리) 정확히 제공 — "추정 없는 현금 기준 실적."
- **현금흐름 보조(표시만):** 감가상각비(D&A) · CapEx를 공시값 그대로 표시. **owner earnings(순이익+D&A−CapEx) 자동계산은 하지 않는다** — 유지/성장 CapEx 구분이 공시에 없어 추정이 되고, "정확한 거울" 원칙에 위배. 재료만 정직하게 주고 판단은 투자자 몫.
- **파생 지표:** 가중 ROE·ROA·부채비율, 투시이익률(투시순이익/투자원금).
- **우선주 처리:** 레벨1(공시값, 보통주/우선주 별도 종목으로) 기본 / 레벨2(우선주 정밀) 향후 / 레벨3(owner earnings) 미구현.
- **데이터:** DART 오픈API(국내 재무제표, 무료) + 토스 기본지표(가능 시). 분기 갱신, 기준 분기 표기.
- **표시(3단 깊이):** ① 대시보드 "내 사업부 실적" 카드 = ROE·PER·PBR 3지표 로테이션(영어, 자동전환, 각자 시장 대비 비교막대 + 좋음/주의 신호, 단정 금지). 위치=보유 종목 근처. ② 투시 상세 = 전체 합산(시각화) + 기업별 기여 목록. ③ 종목 상세 = 개별 단독 재무. 결산·CFO 리포트과 결합 = "버크셔 연차보고서 개인판".
- **세 층위 구분(섞지 않음):** XIRR(시장이 매긴 수익률) / 순자산(내 지갑) / 투시 재무제표(내가 소유한 사업의 실체).
- **ETF는 "제외"가 아니라 "한 단계 더 펼친다"(ETF look-through):** ETF는 회사 재무가 없지만 주식 묶음이므로 펼치면 투시 가능. **L1(V2):** 추종 국가·섹터 지수 밸류에이션(미국 PER 33 / 코스피 22 / 중국 14 / 반도체 섹터 등)으로 "싼 시장/섹터냐" 판단 — ETF 자본배분 도구(개별주=기업 투시, ETF=국가/섹터 밸류). **L2(V3):** 구성종목 펼쳐 가중 PER·ROE(진짜 보유 기업 펀더멘털). 사용자 화면은 동일 — 기업 누르면 기업 투시, ETF 누르면 ETF 투시, 같은 "펀더멘털 카드"(정밀도는 내부에 숨김).
- **범위: V2** (DART 연동 필요). 로직은 명세에 확정.

### 펀더멘털 플래그 — 재무제표가 던지는 질문 (명세 11번, V2~V3)

투시 재무제표 위에 얹는 해석 레이어. "숫자 보여주기"를 넘어 "무엇을 의심하게 하는지" 안내. **단정·진단·투자조언이 아니라 "플래그 + 질문 + 확인처" 형식** — 같은 신호도 좋고 나쁨이 갈리므로 단정하지 않는다(추정 금지·자문업 경계).
- **1급(이익의 진짜 여부 — 핵심):** ① 순이익↑인데 영업현금흐름↓(이익의 질, 분식 1순위) ② 순이익 vs 영업이익 괴리(일회성/영업외) ③ 매출채권이 매출보다 급증(밀어내기) ④ 재고가 매출보다 급증(떨이 임박). 전부 "이 이익이 진짜인가"로 수렴 — 현금흐름표를 넣었기에 가능.
- **2급:** ⑤ 이자보상배율<1(좀비기업).
- **표시:** 종목 상세(개별) + 투시 상세·CFO 리포트(전체 요약, 서한이 문장으로 풀이).
- 데이터는 투시와 동일(DART), 룰 엔진만 추가.

## 8-2-b. 사업부 기여도 분석 — "이 자회사 빼면 내 수익률은?" (V1)

회장이 "이 사업부가 회사 실적을 끌어올리나/까먹나"를 따지는 자본배분의 핵심 도구. 도미노·더리치 어디에도 없는 차별 기능.

- **동작:** 대시보드/수익률 상세의 보유 종목(자회사) 목록에 **체크박스(토글)**. 끄면 그 종목을 제외하고 **XIRR를 실시간 재계산** → "이 조합의 수익률은 X%". 여러 개 켜고 끄며 "미국만 / ETF 빼고 / 손실 종목 빼고" 시뮬레이션.
- **의미 = 기여도:** 전체 XIRR 12%인데 A를 빼니 15%로 오르면 "A가 3%p 깎고 있다", 10%로 내리면 "A가 효자". 어느 자회사가 실적을 끌고 가나/까먹나를 수익률로 직접 확인.
- **계산:** 기존 XIRR 엔진에 *선택된 종목의 이벤트만* 넣어 재계산(엔진 재사용). 종목 제외 시 그 매수금은 **"안 샀다고 치고"(현금 보유로 환산)** 재계산 — "이 종목을 포트폴리오에서 뺐다면 전체가 어땠을까"가 올바른 기여도 해석.
- 범위: XIRR 재계산이라 V1(V0은 전체 수익률부터). 투시 재무제표의 "기업별 기여 목록"(실적 기여)과 짝 — 이쪽은 *수익률* 기여.

## 8-2-c. 자산배분(비중) — 회장이 보는 사업 포트폴리오 (V0~V1)

도넛 차트 + 항목별 비중 목록(토스/도미노식). 단순 "종목 비중"을 넘어 **회장이 자기 사업 포트폴리오의 분산·쏠림을 보는** 도구.

- **분류 4탭(V0~V1 핵심):** **종목별**(집중도 — 한 종목 몰빵?) / **국가별**(미국 몇 %·중국 몇 %, §8-2 ETF 국가 밸류에이션과 연결 — 비싼 시장에 쏠렸나) / **산업별**(반도체 쏠림 등 섹터 위험) / **계좌별**(ISA·연금·일반 어디에 — 세금·용도 관점).
- **도미노 대비 차별:** 도미노는 종목/유형/계좌 3탭. 우리는 **국가별·산업별을 추가** — 가치투자자는 "어느 나라·산업에 쏠렸나"를 종목보다 중시한다.
- **자동 분류:** 종목 마스터의 국가·산업 태그로 자동 집계(사용자가 안 나눔). **ETF의 진짜 노출은 국가/산업 분류에서 드러난다** — 종목별론 "KODEX 미국나스닥 1개"지만 국가별로 펼치면 "미국 50%". 비중 화면이 ETF look-through의 입구.
- **확장:** 유형별(주식/ETF/채권/현금, V1~V2) · 스타일별(가치/성장/배당 — 칭호 시스템 연결, V1~V2) · **투시 기여별**(평가액 비중이 아니라 *실적 기여* — "삼성 비중 37%인데 내 순이익 기여는 60%", V2. 투시 재무제표와 연결).
- 데이터: 종목 마스터 국가·산업 태그(기본 분류는 공짜). 도넛은 빌드 시 차트 라이브러리로 렌더.

## 8-3. 종목 상세 기본지표 — 보유 종목 점검용 (스크리너 아님)

보유한 종목을 눌렀을 때 그 **종목 상세 화면 안에서만** 기본지표를 본다: PER·PBR·ROE·배당수익률·시가총액(티커로 조회).

- **맥락이 핵심:** 이건 "새로 살 종목 고르기"가 아니라 **"내 자회사 점검"**이다. CEO가 자회사 재무를 보는 것과 같아 컨셉과 맞다.
- **금지:** 대시보드 전면 노출 금지(종목 스크리너처럼 보이면 정체성 붕괴). **매수 추천·"이 종목 어때요" 류 절대 금지**(그 순간 트레이딩 앱이 됨, 과매매 유도 = 우리 철학의 반대).
- 범위: 종목 상세와 함께 V1. 데이터는 티커 조회(공짜).

## 8-4. 내 사업부 IR — 보유 종목 공시 피드

내가 보유한 종목의 DART 공시를 모아 피드로 보여준다. **"CEO가 자회사 중요 소식을 받아본다."** 내가 *소유한 회사만* 필터링 → 트레이딩 앱의 "인기 뉴스"와 정반대(살 종목 구경이 아니라 내 사업체 점검).

**공시 유형별 구분 (DART는 유형 코드로 필터 가능):**
- **정기공시**(사업·분기·반기보고서): 주기적 재무제표. → XBRL로 숫자 추출 = 투시 재무제표(§8-2)의 재료.
- **수시공시/주요사항보고**(합병·분할·유상증자·자사주·대규모 계약·소송·최대주주 변경 등): *사건이 터질 때* 나오는 예측 불가 뉴스. **피드의 핵심이자 재방문 트리거의 주인공.** 제목+요약+DART 원본 링크.
- **지분공시**(5%·임원 지분변동): 참고용, 후순위.
- 우선순위: 수시공시 > 정기공시 > 지분공시.

**원문은 가공하지 않는다(단계화):**
- (1) **목록+요약+DART 원본 링크** = 피드(V2). 우리는 큐레이터(내 보유 종목만 빠르게 알림), DART는 원본 보관소. 누르면 DART 원본으로 이동.
- (2) **XBRL 숫자 추출** = 투시 재무제표(V2).
- (3) **AI 요약**("이번 분기 핵심 3줄") = V3, CFO 리포트과 결합. 원문 파싱·AI 비용으로 후순위.
- 수백 페이지 원문을 그대로 노출하지 않는다(아무도 안 읽음 — 토스·뱅샐이 공시를 안 보여주는 이유).

**펀더멘털 플래그와 결합(해석 힌트):** 공시가 *의미하는 바*를 질문으로 던진다 — 유상증자→"지분 희석 가능, 목적·규모 확인", 합병→"구조 변경, 합병비율·목적 확인", 대규모 소송→"우발부채 점검". 단정 아니라 힌트. 흐름: 피드(소식 발견) → 플래그(무슨 의미? 힌트) → DART 링크(원본 확인) → 판단은 본인.

**진입점·노출 위치 (두 곳, 묶는 단위가 다름):**
- **통합 IR 피드**(보유 전체, 시간순): 발견용. 하단 탭(연혁 계열)에서 진입 + **대시보드 알림 배너**("내 사업부 소식 ●3 →", 소식 있을 때만 노출, 토스 "할 일" 자리처럼). 수시공시는 푸시 알림 → 이 피드로 연결. 재방문 트리거의 핵심.
- **종목 상세 화면**(그 회사만 필터): 심화용. 한 자회사를 파고들 때.
- 둘 다 동일 DART 데이터, 묶는 단위만 다름(피드=아침 브리핑 / 종목상세=집중 점검).

- 데이터: DART 오픈API 공시목록 API + XBRL 재무 API(무료). 범위: V2(DART 연동과 함께), AI 요약은 V3.

## 8-5. 거장 포트폴리오 열람 (비전, V2~V3)

레전드 벤치마크(§8)를 한 단계 확장: 거장이 *지금 무엇을 보유하는지*까지 본다.

- **미국 기관(버크셔·ARK 등):** 13F 공시로 분기별 보유 종목·비중·매매가 공개 → 열람 가능. "버핏이 지금 뭘 들고 있나." **탭 구성: 포트폴리오 / 매수 주식 / 매도 주식**(13F의 진짜 가치는 *변화* — 이번 분기 무엇을 샀나/팔았나). 도넛 비중 + 전분기 대비 + "2026 1Q 13F 기준" 표기.
- **내 회사와 구성 비교:** 수익률 비교를 넘어 *포트폴리오 구성*을 같은 형식으로 나란히("버핏은 애플 40%, 나는 분산"). §9 분석카드의 끝판왕 — "이기기"를 넘어 "거장처럼 배우기."
- **한국 한계:** 사모펀드·개인은 공시 의무 없어 포트폴리오 비공개. 국민연금 등 분기 공시만 제한적.
- 데이터: 13F(미국, 공개) 파이프라인 필요. 범위: V2~V3 비전.

## 8-6. 경쟁 앱 검증 기능 (도미노·더리치에서 확인된 UX)

도미노·더리치는 우리 컨셉의 *시장 검증*이자 우리 명세의 *구현 가능성 증거*다. 그들에게 있고 우리에게 가치 있는 것만 선별 채택(광고·다크테마·과열 수익률 리그·초고배당 부각은 제외 — 우리 철학과 반대).

- **백분위 공유 카드(V1):** "투자원금 3천~1억 사용자 중 상위 7%" 같은 **칭호 + 분포 히스토그램 + 공유 버튼**. §9 분석카드의 구체화. 바이럴 엔진(rational.cap 유입). 단 칭호·기준은 우리답게(가치투자 톤).
- **자산 가리기 토글(V1):** 순자산을 ✱✱✱로 블러 처리하는 토글. 큰돈 다루는 앱의 필수 배려(남에게 화면 보일 때). 설정/대시보드 히어로에.
- **배당 캘린더(V1):** 월별 달력에 배당일 표시 + 보유 종목 배당락일·예상금액. 도미노·더리치 둘 다 보유 = 배당 투자자 인기 검증. 우리 배당 이벤트 데이터로 시각화. 단 시가배당률 비정상 고배당(함정) 부각 금지 — 오히려 펀더멘털 플래그 대상.
- **ETF "실질부담비용" 순위(V1~V2, 마찰비용 §8-1과 연계):** 명목 보수율(0.01%)과 실질부담비용(0.1017%)이 다름을 보여주고, 같은 추종 ETF끼리 "어디가 진짜 싼가" 순위. 더리치의 "실질부담비용" = 우리 TER 인사이트의 확장. 보유 ETF의 진짜 비용을 정면에.

## 10. 게이미피케이션

핵심 원칙 — **"더 거래하게"가 아니라 "오래 보유·정기 회고·비용 절감"을 보상한다.** 도미노·토스는 게임으로 매매를 부추겨 수수료를 벌지만, 우리 철학은 정반대다. 모든 보상은 좋은 투자 습관을 향한다.

> **선행 제약(헌법):** 아래 모든 게이미피케이션 요소는 **③ 축하 정책 매트릭스**(결정·규율만 축하, 시장 결과는 절대 축하 안 함)를 통과해야 한다. 운빨을 실력처럼 보이게 하면 거짓이고, 거짓 축하는 신뢰를 무너뜨린다. 심화·시점 매핑: `gamification-honest-roman-v1.md`.

- **회사 설립:** 가입 = 창업 의식 (설립 등기 포함)
- **이벤트 = 경영 행위:** 매수는 "자회사 인수", 입금은 "증자". 기록 자체가 게임 액션이며, 즉각적 체결 연출이 보상이다.
- **기업 정체성은 비위계:** 개인투자자·펀드매니저·패밀리오피스 등 운용 형태에 상하를 두는 등급 사다리는 만들지 않는다. 규모가 아니라 각자의 스타일과 규율을 나란히 인정한다.
- **분기 결산:** 매 분기 결산 이벤트 — 입력이 아니라 *성과 확인 + CFO 리포트 발행* 의식. (이벤트 드리븐이 입력을 담당하므로, 결산은 순수한 보상 이벤트가 된다)

**★ 핵심 3종 (V1~V2):**

- **① 연차보고서 (ENUF Annual Report) — 최강 리텐션:** 설립 후 1년을 채우면 *자기 회사의 연차보고서*가 발행된다(카드/PDF). 올해 XIRR, 투시 실적, 베스트/워스트 사업부, CFO 총평, 표지에 회사명·설립연도. **1년 버틴 사람만** 받는 희소성(리텐션) + SNS 자랑(바이럴) + rational.cap 연례 콘텐츠. 버크셔 연차보고서가 투자계 축제이듯, 개인판 축제.

- **② 투자 스타일 칭호 (다축 — 등급 아닌 색깔):** 수익률(운빨)이 아니라 **행동 패턴**으로 칭호. 가치투자만 떠받들지 않고 여러 길을 나란히 인정(중요 — 편협 방지). 판단 기준은 보유 섹터·회전율·집중도·변동성·보유기간·비용 등 행동 데이터.
  - *가치 계열:* 버핏형 장기보유가 / 그레이엄형 안전마진가 / 멍거형 절제가
  - *성장·기술 계열:* 개척자(파이어니어) / 이노베이터(테크·바이오 집중) / 모멘텀 라이더 / 벤처 캐피탈리스트
  - *스타일 무관:* 인덱서(보글형, 저비용 분산) / 배당 수집가 / 철벽 수비수(방어·현금) / 글로벌 분산가
  - **위아래가 아니라 옆으로 나란히** — 개척자가 버핏형보다 못한 게 아니라 다른 길. "투자는 한 가지 정답이 아니다"(rational.cap 철학). 단 모멘텀 계열도 *잘 타는 것*은 인정하되 *자주 갈아타는 것*은 마찰비용으로 경고(과매매 방지). "너는 개척자인데 버핏형처럼 자주 판다" 같은 스타일-행동 불일치 자각도 가능.

- **③ 결산 스트릭:** 분기마다 결산(CFO 리포트 수령+회고)하면 🔥스트릭 누적("8분기 연속 결산"). 끊기면 아까워 계속 옴. *좋은 습관(정기 회고)을 게임화* — 건강한 중독.

**보조 (V2~):**
- **올해의 결정:** 연말에 베스트/워스트 매수 회고 + **"안 팔고 버틴 것"도 칭찬**("3월 폭락 때 인내했습니다"). 손실회피가 아니라 인내를 보상.
- **사업부 MVP:** 분기별 투시 이익 기여 1위 종목("이번 분기 MVP 사업부: 삼성전자").
- **설립기념일:** 창립 N주년 축하 + 여정 요약(감정적 애착).

## 11. 킬러 기능: CFO 리포트 (분기 경영 보고)

사용자는 지주회사의 **회장(CEO)**이고, AI는 그 밑의 **CFO**다. CFO가 분기마다 회장께 올리는 경영 보고 — 회계 실무의 **MD&A(경영진단의견서)** 성격이다. 숫자 뒤의 이야기를 풀어 보고하되, *결정은 회장 몫*이다.

- **이름:** 화면 표기는 **"CFO 리포트"**(직관), 본문은 "회장님, 이번 분기를 보고드립니다" 톤(컨셉 각인 — 사용자를 회장으로 대접). 알림: "CFO가 분기 보고를 올렸습니다 →".
- **역할 경계(중요 — 자문업 방지):** CFO는 **보고·해석·리스크 환기·직언까지.** "이 종목 사라/팔아라" 같은 매수·매도 추천은 하지 않는다(그건 회장의 결정). MD&A는 사실과 해석이지 투자 지시가 아니다 — 이 경계가 자문업 리스크를 컨셉 차원에서 차단한다. "왜 추천 안 해?"가 아니라 "CFO는 보고하고 회장이 결정한다"가 자연스럽다.
- **비용 효율(핵심):** **계산은 엔진(코드)이, 서사화만 AI가.** 수익률·플래그·투시 수치는 우리 엔진이 계산(공짜·정확)하고, AI는 그 수치를 문장으로 풀기만 한다(수치 주입, 모델은 서술만 — AI에 계산 시키면 토큰↑·오류↑). **분기 1회 배치 생성**이라 사용자가 늘어도 비용은 천천히 증가(1인 연 4통). 중급 모델로 충분(창의력보다 정해진 톤 정리). 시세 API와 달리 비용 부담 낮음.
- **내용:** 이번 분기 XIRR·벤치마크 대비·투시 실적 요약 + 보유 기업 펀더멘털 플래그 환기("A사 현금흐름 주의 — 검토 안건") + 과매매 시 멍거 톤 직언("마찰비용이 복리를 갉았습니다"). 칭찬만이 아니라 *행동을 교정하는* 보고가 차별점.
- **핵심 가치 — 단순 요약이 아니라 개인화 진단:** "당신의 XIRR은 18%" 같은 숫자 풀이는 무료로도 보이므로 그 자체론 약하다. CFO 리포트의 진짜 값은 **내가 혼자선 못 보는 것을 짚어주는 것** — ① 맹점 발견("보유 A사는 3분기 연속 현금흐름이 나빠지는데 이익은 늘었습니다"), ② 행동 교정("지난 4분기 오를 때 사고 떨어질 때 팔았습니다 — 평균 진입가가 시장가보다 높습니다"), ③ 스타일 일관성("성장주 투자자인데 방어주를 늘렸습니다 — 의도한 변화인가요?"). 코칭이지 칭찬이 아니다.
- **생성 방식 — 요청 시(on-demand) + 캐시:** 미리 만들지 않는다. 분기말엔 알림만(공짜), 사용자가 열람을 누를 때 1회 생성 후 저장(재조회 무료). 휴면 사용자에겐 생성 안 함 → 비용이 실제 열람자에만 비례. (수익모델 13번)
- **결산과의 분리:** **결산(분기 숫자 정리)은 엔진이 계산해 무료**로 보여주고, 그 위에 **CFO 리포트(AI 서사)가 선택적으로 얹힌다.** 숫자는 누구나, AI 글은 누른 사람만.
- **활용:** PDF 저장 / 가족 공유 / SNS(Threads, rational.cap) 공유 → 지적 성취 기반 바이럴(영향력 확산).

## 13. 수익 모델

**전제(비전 2번):** 1차 목표는 영향력(철학 확산)이며 수익은 후순위다. 따라서 **초기엔 핵심 기능도 최대한 무료로 개방**해 확산·검증을 우선하고, 과금은 영향력이 쌓이고 운영비 감당이 필요할 때 도입한다. 과금 라인을 미리 그어두되 *성급히 닫지 않는다.*

- **무료(확산 우선):** 회사 설립 · 이벤트 기록 · 순자산 조회 · **결산 숫자**(XIRR·자산변화·종목 성적 — 엔진 계산이라 토큰 0, 공짜로 줘도 됨) · 기본 벤치마크. 초기엔 CFO 리포트도 무료 개방해 가치 검증.
- **추후 Premium(검증 후):** **CFO 리포트의 개인화 진단**(단순 요약이 아니라 *내가 못 보는 맹점·반복 실수·행동 교정* — 혼자 못 하는 코칭이 진짜 돈값) · 연차보고서 · 투시 재무제표(V2) · PME 벤치마크.
- **수익 사다리(영향력 확보 후):** 구독(메인, 이해상충 없음) → 투권인 **계좌개설 리워드**(거래 수수료 아님 — 과매매 이해상충 방지) → 자문(V2~V3 정점).

**AI 비용 설계 — 요청 시 생성(on-demand) + 캐시:**
- CFO 리포트·연차보고서는 **미리 생성하지 않는다.** 분기/연말엔 알림만(공짜 텍스트), 사용자가 *열람을 누를 때 1회 AI 생성 후 저장(캐시)* — 재조회는 무료, 휴면 사용자에겐 생성 안 함. 비용이 "실제 열람자 수"에만 비례.
- **계산은 엔진(코드, 공짜), AI는 서사화만.** 연차보고서는 90%가 숫자·차트·템플릿 조립(공짜)이고 AI는 "총평 1페이지"만 — 분기 CFO 리포트들을 재료로 요약해 더 절감.

**광고는 하지 않는다.** 증권사 광고는 "더 거래하라"를 부추겨 *마찰비용 반대*라는 우리 철학과 이해상충하고, "차분한 경영 콘솔" 컨셉과 rational.cap의 신뢰를 훼손한다(도미노·더리치·토스가 광고로 수익화하는 것과의 결정적 차별점). 단기 수익보다 컨셉·신뢰·영향력을 택한다.

## 15. 제품 로드맵

- **V0:** 개인 투자회사 — 설립 등기 · 이벤트 기록 · XIRR · CEO 대시보드 (단일 계좌, PWA). 대시보드는 확장 가능한 카드 그리드 구조로 설계. **운영 모드 ledger·challenge 지원**(장부=자기·시장 경쟁 / 챌린지=오늘 고정, 시장·과거의 나 경쟁). mode 필드 + 오늘 고정 규칙은 데이터·로직에 심되 리그 UI는 V1. **모드는 회사 단위 고정(전환 불가) — 다른 모드를 원하면 새 회사를 설립한다. 한 사용자가 여러 Holding(회사)을 거느릴 수 있다(회장 컨셉). 단 V0 화면은 회사 1개만 노출, 멀티 회사·전환 UI는 V1.**
- **V0.5 (fast-follow):** 토스 시세 연동 + 일일 변동 알림 · CFO 리포트 · PME(vs 시장) · **분석·비교 카드 팩** — vs 실존 투자회사(버크셔·국민연금 공개 수익률과 내 XIRR 비교, "버핏을 이겼습니다"), 운용 스타일 진단(회전율·집중도·현금비중·보유기간), 자기 분석 카드.
- **V1:** 멀티 계좌 — ISA · 연금 · 복수 증권사 (Account 레이어 활성화) · **vs 다른 유저 백분위 비교(유저 수 임계치 도달 시 활성화)** · **챌린지 유저 간 순위 UI(검증 배지 표시)** · **멀티 회사(여러 Holding) 전환 UI** + **회사명 변경**(대시보드 상단 회사명 탭) · **사업부 기여도 분석**(자회사 체크박스로 빼고 XIRR 재계산 — "이 종목 빼면 수익률 몇 %?", §8-2-b) · 대시보드 꾸미기(카드 켜고 끄기) · **투권인 계좌개설 리워드** — 창업자 개인 투권인 위촉(자본금 0)으로 온보딩 시 증권 계좌 개설 연결, 트래픽 성장 후 증권사-법인 제휴로 전환. **거래 기반 수수료는 받지 않는다(과매매 억제 컨셉과의 이해상충 방지 원칙).** · **백분위 공유 카드**(칭호+분포+공유, 바이럴) · **자산 가리기 토글**(✱✱✱ 블러) · **배당 캘린더**(보유 종목 배당락일·예상금) · **ETF 실질부담비용 순위**(§8-1) — 도미노·더리치에서 검증된 UX 선별 채택(광고·다크테마·과열 수익률 리그·초고배당 부각은 제외)
- **V2:** Family Holding — 배우자/자녀 계좌 묶음, 가족 그룹 통합 · **주니어 사업부(용돈 401k)** — 자녀 계좌 = 자회사, 용돈 = 정기 증자, 부모 매칭 룰("아이가 5만원 저축하면 부모가 5만원 매칭"). 실제 송금은 기존 금융앱이 담당하고 본 서비스는 기록·매칭 알림·연혁을 담당. 성인이 되는 날 회사 연혁 전체를 물려주는 세대 전승 서사 = 최강 리텐션 장치 · **부동산 사업부**(Account 유형 REAL_ESTATE 추가, 국토부 실거래가 연동. 증권 Event 모델과 분리된 평가 파이프라인 — XIRR 정밀도를 오염시키지 않도록 별도 표기) · **★ 투시 재무제표(look-through financials)** — DART 재무 연동으로 내 지주회사의 실제 P&L·B/S·가중 ROE를 보유 지분만큼 합산(명세 `lookthrough-financials-spec-v1.md`). 결산·CFO 리포트과 결합해 "버크셔 연차보고서 개인판" 완성 · **내 사업부 IR**(보유 종목 DART 공시 피드, 재방문 트리거) · **MP(모델 포트폴리오) 자문 1단계** — 유사투자자문 신고(불특정 다수 대상 동일 콘텐츠, 자본금 0) 또는 등록 자문사 B2B2C 제휴로 MP 구독 기능 검증. 창업자의 앱 내 공개 XIRR이 트랙레코드가 된다.
- **V3:** 완전 자동화 + 네이티브 앱 — API·마이데이터 동기화, KMP 앱 · **라이브(live 모드)** — 증권사 연동 시 신규 Holding(라이브 회사)이 자동 생성되고 과거 거래가 딸려와 채워진다. 연동된 본인 계좌만 존재 검증되어 진짜 순위. 가족은 각자 연동해 그룹 합산(V2 Family Holding와 연계) · **거장 포트폴리오 열람(13F 기반)** — 버크셔 등 미국 기관의 공시 보유 종목·비중을 보고, 내 회사와 포트폴리오 구성을 같은 형식으로 비교("버핏은 애플 40%, 나는 분산"). 수익률 비교(§8)를 넘어 "거장처럼 배우기" · **MP 자문 2단계** — MP 구독 매출이 등록 비용(자본금 약 2.5억)을 정당화하면 **BLH Capital 명의로 투자자문업 등록**, 1:1 맞춤 자문으로 확장. **투자일임업은 하지 않는다** — "사용자가 CEO"라는 제품 비전과 모순되며(판단 주권은 항상 사용자에게), 고객 자산 직접 운용의 분쟁·컴플라이언스 리스크를 배제한다.

> **비교는 3층 구조다: ① vs 시장(PME) → ② vs 실존 투자회사 + 스타일 → ③ vs 다른 유저.** ①②는 공개 데이터라 유저 0명이어도 작동하고, ③만 유저 규모가 필요하므로 마지막이다.

> **수익화 사다리: 구독(거울) → 계좌개설 리워드(투권인, 거래수수료 배제) → MP 자문(유사자문/제휴 → BLH 자문업 등록).** 각 단계는 앞 단계의 트래픽·신뢰를 먹고 자라며, 어느 단계에서도 "정확한 거울"이라는 본질과 이해상충하지 않는다.

## 🛠️ 부록: 백엔드 DB 스키마 설계 프롬프트 (v0.5 — 이벤트 드리븐 모델)

### 🌐 English Prompt
> **Role:** Senior Database Architect and Backend Engineer.
>
> **Task:** Generate a Supabase (PostgreSQL) schema with Row Level Security policies, based on the ENUF PRD v0.6, plus TypeScript types for a Next.js app.
>
> **Requirements:**
> 1. Implement `holdings`, `accounts`, and `events` tables. Do NOT create separate cash_flows/transactions tables — a single `events` table with a `type` enum ('BUY','SELL','DIVIDEND','DEPOSIT','WITHDRAWAL') covers all five investment actions.
> 2. **Events table:** `type`, `symbol` (nullable — only BUY/SELL/DIVIDEND), `quantity` (nullable — only BUY/SELL), `price_or_amount`, `fee_and_tax` (numeric, default 0), `date` (required), `account_id` FK.
> 3. **Account layer (critical):** `events` reference `accounts(id)`; `accounts` reference `holdings(id)`. Never attach events directly to a holding. Auto-create one default account ('main', GENERAL) per holding on signup.
> 4. **Accounts table:** include `account_type` enum ('GENERAL','ISA','PENSION','OVERSEAS') and `commission_rate` (numeric). Tax rules are derived from `account_type`; commission is per-account via `commission_rate`. For v0 implement GENERAL tax logic only (dividend 15.4% + transaction tax); leave hooks for the others.
> 5. **Holdings table:** include founding-registration fields — `founded_at`, `initial_capital`, `initial_valuation` (one-time onboarding snapshot; performance is computed only from founding date forward). Also a `mode` enum ('ledger','challenge','live') determining data-verification level and competition scope (v0 uses ledger/challenge; live is V3). A user can own multiple holdings with different modes; mode is fixed per holding (no in-place switching).
> 6. **Positions are derived, not stored as source of truth:** provide a Postgres view (or materialized view) `positions` computing per-account per-symbol quantity and average cost from buy/sell events. Users must never edit positions directly.
> 7. **Current valuation** = Σ(position quantity × live price from external quote feed — Toss Securities Open API) + cash balance, where cash balance = deposits − withdrawals + dividends + sell proceeds − buy costs − Σ(fee_and_tax), all derived from events. The schema stores no live prices. Keep tax rates in a config table/constant, not hardcoded.
> 8. Add Supabase RLS so each user can only read/write their own holdings, accounts, and events.
> 9. Provide clean, production-ready SQL migrations and TypeScript types.

### 🇰🇷 한글 번역
> **역할:** 시니어 DB 아키텍트 / 백엔드 엔지니어
>
> **작업:** ENUF PRD v0.6 기반 Supabase(PostgreSQL) 스키마 + RLS 정책 + Next.js용 TypeScript 타입 생성.
>
> **요구사항:**
> 1. `holdings`, `accounts`, `events` 테이블 구현. cash_flows/transactions를 분리하지 말고, `type` enum('BUY','SELL','DIVIDEND','DEPOSIT','WITHDRAWAL')을 가진 단일 `events` 테이블로 다섯 가지 투자 행동을 모두 커버.
> 2. **events:** `type`, `symbol`(nullable — 매수/매도/배당만), `quantity`(nullable — 매수/매도만), `price_or_amount`, `fee_and_tax`(numeric, 기본 0), `date`(필수), `account_id` FK.
> 3. **계좌 레이어(핵심):** events는 accounts(id)를, accounts는 holdings(id)를 참조. 이벤트를 holding에 직접 붙이지 말 것. 가입 시 holding마다 기본 계좌('main', GENERAL) 1개 자동 생성.
> 4. **accounts:** `account_type` enum('GENERAL','ISA','PENSION','OVERSEAS')과 `commission_rate`(numeric) 포함. 세금 규칙은 account_type에서 파생, 수수료는 계좌별 commission_rate로 적용. v0는 GENERAL 세금 로직만 구현(배당 15.4% + 거래세), 나머지는 훅만 남겨둘 것.
> 5. **holdings:** 설립 등기 필드 포함 — `founded_at`, `initial_capital`, `initial_valuation` (1회성 온보딩 스냅샷. 성과는 설립일 이후만 계산).
> 6. **포지션은 파생 데이터:** 매수/매도 이벤트에서 계좌별·종목별 수량과 평균단가를 계산하는 Postgres 뷰(또는 materialized view) `positions` 제공. 사용자가 포지션을 직접 수정할 수 없어야 함.
> 7. **현재 평가액** = Σ(보유수량 × 외부 시세 — 토스증권 Open API) + 현금 잔고. 현금 잔고 = 입금 − 출금 + 배당 + 매도대금 − 매수대금 − Σ(fee_and_tax) (전부 이벤트에서 파생). 스키마는 실시간 시세를 저장하지 않음. 세율은 하드코딩하지 말고 config 테이블/상수로 분리.
> 8. 사용자가 자신의 holdings·accounts·events만 읽고 쓸 수 있도록 Supabase RLS 적용.
> 9. 프로덕션 적용 가능한 SQL 마이그레이션과 TypeScript 타입 제공.

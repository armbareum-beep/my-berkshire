# Phase 0 Research: 부동산 사업부 실투자금 수익률·순자산·LTV

명세에 미해결(NEEDS CLARIFICATION) 항목 없음 — 표시 위치·지표 이름·기존 수익률과의 관계는 사전 대화에서 확정됨. 아래는 설계 결정 기록.

## D1. 실투자금 수익률의 분모 — 취득원가 기준 vs 현재가치 기준

- **Decision**: 분모(실투자금) = **취득원가 기준** = (취득가 + 부대비용 + 자본투입) − 대출잔액. 분자는 이미 이자 차감된 누적손익(`gain`).
- **Rationale**: 기존 자산수익률(`gain/cost`)과 분자를 공유해 "같은 손익, 분모만 내 돈"으로 레버리지 효과가 직관적으로 대비된다. 수익률의 분모는 투입자본이어야 일관(현재 순자산은 별도 *순자산* 지표로 표시).
- **Alternatives**: 현재가치 기준(현재평가−대출잔액) — "지금 내 몫 대비 수익"은 회계적으로 retroactive해 부정확. 순자산 스냅샷 용도로만 채택(D2).

## D2. 순자산(netEquity)과 LTV의 분모 — 현재 평가액

- **Decision**: 순자산 = (보유 물건 **현재 평가액** 합) − 대출잔액. LTV = 대출잔액 / 현재 평가액.
- **Rationale**: "지금 내 몫"·"지금 빚 비중"은 현재 시점 스냅샷이어야 의미. 기존 `netWorth(assets, debt)`·`leverageRatio(assets, debt)`([liabilities.ts](../../src/lib/finance/liabilities.ts)) 재사용.
- **Alternatives**: 취득원가 기준 LTV — 시세 변동을 못 담아 위험 신호로 부적합.

## D3. 대출잔액을 원차입액 근사로 사용

- **Decision**: 실투자금 분모의 대출잔액은 **현재 잔액**을 그대로 사용(원차입액 대신).
- **Rationale**: 앱은 대출의 현재 잔액만 보유. 모기지 원금 상환은 느려 근사 오차가 작고, 개인 가계부 정확도로 충분. spec 012도 동일 가정(이자=잔액×이율).
- **Alternatives**: 원차입액 별도 입력 — 입력 부담↑, 신규 컬럼 필요 → 가치 대비 과함(원칙 III·IV).

## D4. 표시 위치 — 상세 페이지 한정, 홈 카드 불변

- **Decision**: 실투자금 수익률·순자산·LTV는 `/real-estate` 상세(`ManualAssetsSection`)에만. 홈 통합 카드(`DivisionCard`)는 기존 자산수익률 한 줄 유지.
- **Rationale**: 홈 통합 카드에 부동산 줄만 지표가 늘면 다른 사업부 줄과 시각적으로 들쭉날쭉(원칙 IV 절제). 레버리지 맥락은 상세에서 충분.
- **Alternatives**: 홈 카드에도 노출 / 통합 요약 한 줄 — 사용자가 상세 한정으로 확정.

## D5. 임대료 없는 거주용·차익형 처리

- **Decision**: "수익률"은 임대수익을 전제하지 않는다. 분자 = 평가차익 + 임대순익(0 가능) + 매도차익 − 이자. 임대 0이어도 평가차익 − 이자로 정상 산출.
- **Rationale**: 사용자 피드백 — 임대료 없이 시세차익·거주용으로 보유하는 경우가 많다. 이름을 "실투자금 수익률"로 정해 income 함의를 제거(ROE 미채택).
- **Alternatives**: ROE/자기자본수익률 — "리턴=수익" 오해 소지로 기각.

## D6. 산출 불가(엣지) 처리

- **Decision**: 실투자금 ≤ 0 → 실투자금 수익률 `null`("—"). 평가액 ≤ 0 → LTV `null`("—"). 대출 0 → 지표 묶음 전체 숨김.
- **Rationale**: 가짜 숫자·∞ 방지(원칙 II). 기존 `leverageRatio`는 자산≤0에서 Infinity 반환하므로, UI 진입 전 `marketValue > 0` 게이트로 null 처리.
- **Alternatives**: 0 또는 Infinity 표시 — 오해 유발, 기각.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 거장 포트폴리오가 항상 최신이다 (Priority: P1) 🎯 MVP

사용자가 거장 포트폴리오 화면에 들어갔을 때 "6개월 전 데이터"가 아니라 가장 최근 분기 13F 기준 포트폴리오를 본다. 분기마다 시스템이 자동으로 SEC EDGAR에서 최신 13F를 수집·파싱·저장한다. 사용자는 아무것도 할 필요 없다.

**Why this priority**: 지금은 하드코딩된 정적 데이터라 점점 낡아진다. 이게 해결되지 않으면 다른 기능(변화 감지, 비교)도 의미 없다.

**Independent Test**: 거장 포트폴리오 화면에서 "버크셔 해서웨이" 선택 시 표시 기준일이 직전 분기 이내이고, 실제 13F 공개 데이터와 보유 종목 목록이 일치하는지 확인.

**Acceptance Scenarios**:

1. **Given** SEC EDGAR에 최신 13F가 제출된 상태, **When** 자동 수집이 실행되면, **Then** DB에 해당 거장의 보유 목록(종목 코드, 주수, 신고 시가)이 저장된다.
2. **Given** CUSIP이 내부 티커 코드와 다른 경우, **When** CUSIP→티커 매핑 단계를 거치면, **Then** 내부 종목 코드로 정규화되어 저장된다.
3. **Given** 기존 데이터가 있는 상태에서 새 분기 13F가 수집되면, **When** 화면을 열면, **Then** 가장 최근 분기 데이터가 표시되고 이전 분기 데이터도 히스토리로 보관된다.
4. **Given** 특정 거장의 13F가 아직 공개되지 않은 경우, **When** 수집이 실행되면, **Then** 기존 최신 데이터를 유지하고 오류 없이 종료된다.

---

### User Story 2 - 거장이 이번 분기 뭘 샀고 팔았는지 본다 (Priority: P2)

분기가 바뀌면 거장 포트폴리오 화면에 "이번 분기 변화" 섹션이 나타난다. 버핏이 이번 분기 새로 산 종목, 늘린 종목, 줄인 종목, 완전히 판 종목을 한눈에 본다.

**Why this priority**: 거장의 매매 변화가 핵심 정보다. 단순 보유 목록보다 "이번에 뭘 샀냐"가 훨씬 가치 있다.

**Independent Test**: 거장 선택 → 변화 탭에서 신규 매수·추가·축소·청산 종목이 각각 구분되어 표시되고, 전 분기 보유 수량과 비교한 변화량이 올바른지 확인.

**Acceptance Scenarios**:

1. **Given** 전 분기와 이번 분기 13F 데이터가 모두 있는 경우, **When** 변화 비교를 실행하면, **Then** 신규(전 분기 없음 + 이번 있음), 추가(수량 증가), 축소(수량 감소), 청산(전 분기 있음 + 이번 없음)으로 분류된다.
2. **Given** 처음 수집된 거장(비교 대상 전 분기 없음), **When** 변화 화면을 열면, **Then** "이전 분기 데이터 없음" 안내와 함께 전체 보유 목록만 보여준다.
3. **Given** 사용자가 내 보유 종목도 있는 경우, **When** 거장이 그 종목을 새로 샀다면, **Then** "내 보유" 배지가 변화 목록에서도 표시된다.

---

### User Story 3 - 여러 거장 최신 현황을 한 화면에서 전환한다 (Priority: P3)

현재 정적으로 지원하는 모든 거장에 대해 자동 수집이 적용된다. 거장 선택 아바타는 유지하되, 각 거장의 "마지막 업데이트" 날짜가 표시된다.

**Why this priority**: P1·P2 완료 후 자연스러운 확장. 구현 추가 비용이 낮고 다 지원해야 의미 있다.

**Independent Test**: 거장 목록에서 여러 명을 전환하면서 각각 기준일이 다르게 표시되고, 기준일이 직전 분기 이내인지 확인.

**Acceptance Scenarios**:

1. **Given** 복수의 거장이 등록된 경우, **When** 거장 아바타를 전환하면, **Then** 각 거장 고유의 최신 13F 기준일과 보유 목록이 표시된다.
2. **Given** 특정 거장의 13F 수집이 실패한 경우, **When** 그 거장 선택 시, **Then** 마지막 성공 수집 데이터와 함께 "업데이트 실패" 안내가 표시된다.

---

### Edge Cases

- 13F 제출 기한(분기 종료 후 45일)이 지났는데 미제출인 경우 — 이전 데이터 유지, 경고 없음.
- CUSIP→티커 매핑이 실패하는 경우(상장폐지, 합병 등) — 해당 종목은 "Unknown(CUSIP)"으로 표시, 나머지는 정상 표시.
- 동일 종목이 여러 주식 클래스(예: A/B 클래스)로 분리된 경우 — 별도 행으로 표시.
- 13F에는 공매도 포지션 포함 안 됨 — 실제 보유와 다를 수 있음을 화면에 명시.
- 한국 기관은 13F 대상 아님 — 미국 SEC 등록 기관만 지원, 범위 화면에 명시.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 등록된 거장 목록에 대해 분기별로 SEC EDGAR에서 13F-HR 데이터를 자동 수집해야 한다.
- **FR-002**: 수집된 CUSIP 코드를 내부 종목 티커 코드로 변환하여 저장해야 한다.
- **FR-003**: 분기별 보유 데이터를 히스토리로 보관하며, 최신 분기를 기본으로 표시해야 한다.
- **FR-004**: 직전 분기 대비 보유 변화(신규·추가·축소·청산)를 계산하여 화면에 노출해야 한다.
- **FR-005**: 거장 포트폴리오 화면에 데이터 기준일(분기 및 제출일)을 표시해야 한다.
- **FR-006**: 사용자 보유 종목과 거장 보유 종목의 겹침 "내 보유" 표시를 유지해야 한다.
- **FR-007**: 수집 실패 시 기존 데이터를 유지하고 조용히 실패해야 한다(사용자 흐름 차단 없음).
- **FR-008**: 13F에 취득가가 없으므로 수익률·XIRR 비교는 제공하지 않으며, 이 한계를 화면에 명시해야 한다.

### Key Entities

- **거장(Legend)**: 이름, 기관명, SEC CIK 번호(EDGAR 조회 키), 아바타 이미지 키.
- **13F 스냅샷**: 거장 ID, 분기(연도·Q), 제출일, 총 신고 시가 합계. 분기별 1건.
- **13F 보유 항목**: 스냅샷 ID, CUSIP, 내부 티커(매핑 성공 시), 종목명, 보유 주수, 신고 시가, 투자 유형(Sole/Shared/None).
- **CUSIP-티커 매핑 캐시**: CUSIP, 티커, 종목명, 마지막 확인일. 재조회 최소화용.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 분기 13F 공개 후 48시간 이내에 거장 포트폴리오 화면에 최신 데이터가 반영된다.
- **SC-002**: CUSIP→티커 매핑 성공률 90% 이상(신고 시가 기준 상위 종목 기준).
- **SC-003**: 거장 포트폴리오 화면 로딩이 현재 대비 동등하거나 빠르다(DB 기반 전환으로 실시간 외부 호출 없음).
- **SC-004**: 분기 변화 섹션에서 신규·추가·축소·청산이 빠짐없이 올바르게 분류된다.
- **SC-005**: 13F 수집 실패 시 사용자가 앱 사용 중단 없이 이전 데이터를 정상적으로 본다.

## Assumptions

- 지원 대상은 현재 `legends.ts`에 하드코딩된 거장과 동일한 범위로 시작한다.
- SEC EDGAR API는 공개 무료 접근 가능하며 별도 인증 불필요하다.
- CUSIP→티커 매핑은 SEC EDGAR 검색 또는 공개 매핑 데이터를 활용한다.
- 13F 수집은 서버사이드 스케줄러로 분기별 1회 자동 실행한다.
- 거장 목록 추가·삭제 UI는 이 스펙 범위 밖이다(코드 레벨에서만 관리).
- 미국 SEC 등록 기관만 지원한다(한국 기관은 13F 대상 아님).
- 기존 LegendExplorer 컴포넌트를 기반으로 UI를 확장한다.

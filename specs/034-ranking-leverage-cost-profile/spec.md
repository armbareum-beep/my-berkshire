# Feature Specification: 랭킹 지표 확장(저레버리지·저비용) + 리더보드 프로필 바텀 시트

**Feature Branch**: `034-ranking-leverage-cost-profile`
**Created**: 2026-07-04
**Status**: Draft
**Input**: User description: "랭킹 점수에 저레버리지·저비용 지표 추가(자산금액·절대수익률은 제외), 리더보드 행 클릭 시 프로필 바텀시트(지표 분해+연혁) 신설, 033 FR-013 랭킹 채점 잠금 해제"

## 개요

032에서 도입한 5지표 랭킹(보유기간·역발상·시장대비·분산·적립)에 저레버리지·저비용 2개 지표를 더해 7지표로 확장한다. 또한 리더보드 행을 눌러 다른 지주회사의 점수 분해와 회사 연혁을 볼 수 있는 프로필 바텀시트를 신설한다. 자산 금액과 절대 수익률은 어떤 형태로도 랭킹·프로필에 노출하지 않는다.

**명시적 결정**: 자산 금액·절대 수익률(XIRR 절대값)은 랭킹 지표에 추가하지 않는다.
- 프라이버시: 순자산 규모가 공개 리더보드에 노출되는 것은 이 앱의 비공개 원칙과 맞지 않는다.
- 운·출발선 보상 문제: 절대 수익률은 시장 국면·매수 타이밍의 운, 초기 자본 규모의 차이를 그대로 반영해 "조작하기 어려운 행동 규율"이라는 032의 설계 원칙과 어긋난다.
- 033 헌법 충돌: 게이미피케이션 헌법(정직한 로망)은 시장 결과(수익·평가액)를 축하·평가 트리거로 쓰는 것을 금지한다. 절대 수익률을 랭킹 점수에 넣는 것은 이 금지와 정면 충돌한다.
- 대안: 시장 국면 대비 상대 성과는 기존 "시장 대비 성과" 지표(내 XIRR − 코스피 PME XIRR)로 이미 반영되어 있어 중복 도입할 필요가 없다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 저비용·저레버리지도 규율 점수에 반영된다 (Priority: P1)

지금까지는 보유기간·역발상·시장대비·분산·적립만 채점됐다. 부채를 끌어 쓰거나 잦은 매매로 수수료·세금을 많이 내는 행동은 점수에 드러나지 않았다. 이제 부채 의존도가 낮을수록, 마찰비용(수수료·세금)이 낮을수록 총점이 올라간다.

**Why this priority**: 032가 처음부터 의도했던 "조작하기 어려운 행동 규율" 세트에서 레버리지·비용 규율이 빠져 있었던 공백을 메우는 핵심 확장이다.

**Independent Test**: 무차입·저비용 계정과 고차입·고비용 계정을 각각 랭킹에 방문시켜 저레버리지·저비용 점수와 총점 차이를 확인하면 단독 검증 가능.

**Acceptance Scenarios**:

1. **Given** 부채가 없는(0원) 지주회사, **When** 랭킹을 방문하면, **Then** 저레버리지 지표가 100점으로 산출된다.
2. **Given** 부채/자산 비율이 40% 이상인 지주회사, **When** 랭킹을 방문하면, **Then** 저레버리지 지표가 0점에 근접한 값으로 산출된다.
3. **Given** 마찰비용/투입원금(drag)을 산출할 수 없는 상태(투입원금 0 등), **When** 랭킹을 방문하면, **Then** 저비용 지표는 50점으로 산출되고 "데이터 부족"이 표시된다.
4. **Given** 부채는 0보다 큰데 현재 평가액(시세) 조회에 실패한 상태, **When** 랭킹을 방문하면, **Then** 저레버리지 지표는 50점으로 산출되고 "데이터 부족"이 표시된다.

---

### User Story 2 - 리더보드 행을 눌러 다른 회장님의 프로필을 본다 (Priority: P2)

지금까지 리더보드는 이름과 총점만 보여주고 행을 눌러도 아무 일도 일어나지 않았다. 이제 행을 누르면 그 지주회사의 등급, 7개 지표 막대, 설립 연차, 회사 연혁을 볼 수 있는 바텀시트가 열린다. 단, 자산 금액·보유 종목·XIRR 절대값은 시트 어디에도 없다.

**Why this priority**: 랭킹이 숫자 나열에 그치지 않고 다른 지주회사의 "걸어온 길"을 볼 수 있어야 리더보드가 서사로 이어진다. 단 비공개 원칙은 절대 원칙이라 지표 확장(P1)보다 아래이지만, 이 기능의 핵심 신규 인터랙션이다.

**Independent Test**: 임의의 타인 행을 클릭해 시트를 열고, 시트 내 모든 텍스트를 훑어 금액·종목명·XIRR 수치가 없는지 확인하면 단독 검증 가능.

**Acceptance Scenarios**:

1. **Given** 리더보드에 표시된 임의의 행, **When** 그 행을 탭하면, **Then** 바텀시트가 열리고 지주회사명·등급 배지·총점·7개 지표 막대·설립 N년차·연혁이 표시된다.
2. **Given** 열린 프로필 바텀시트, **When** 표시된 모든 내용을 확인하면, **Then** 원화 금액, 보유 종목명, XIRR 절대 수치가 어디에도 없다.
3. **Given** 아직 저레버리지·저비용이 산출되지 않은 구버전(`score_version=1`) 행, **When** 그 행의 프로필 시트를 열면, **Then** 해당 두 지표가 "산출 대기"로 표시된다.
4. **Given** 연혁 데이터(`milestones`)가 없는 구버전 행, **When** 프로필 시트를 열면, **Then** 연혁 영역에 "연혁 준비 중"이 표시된다.

---

### User Story 3 - 방문만으로 최신 채점 방식으로 자연 갱신된다 (Priority: P3)

구버전 방식(5지표)으로 저장된 점수는 본인이 대시보드나 랭킹을 열 때 별도 조치 없이 7지표로 다시 계산되어 저장된다.

**Why this priority**: 마이그레이션 배치 없이 기존 032 참여자 전원을 자연스럽게 신버전으로 옮기기 위한 전환 경로. 사용자 대면 신규 가치는 없지만 무중단 전환에 필요하다.

**Independent Test**: `score_version=1`(또는 null)로 저장된 본인 계정으로 로그인해 `/dashboard` 또는 `/ranking`을 방문한 뒤 DB 행이 `score_version=2`와 7지표 값으로 갱신됐는지 확인.

**Acceptance Scenarios**:

1. **Given** `score_version` 이 1(또는 null)인 본인 랭킹 행, **When** `/dashboard` 또는 `/ranking`을 방문하면, **Then** 방문 직후 해당 행이 `score_version=2`, `leverage_score`·`cost_score`·`founded_at`·`milestones`가 채워진 상태로 갱신된다.
2. **Given** 타인의 구버전 행, **When** 그 행을 리더보드에서 열람만 하면(본인 방문 아님), **Then** 그 행은 재계산되지 않는다(자기 갱신만 발생, 열람으로는 갱신되지 않음).

---

### Edge Cases

- 부채는 있는데 자산 평가액 자체가 0이면? → 저레버리지 데이터 부족(50점)으로 처리(0으로 나누기 방지).
- `founded_at`이 없는 매우 오래된(기능 도입 이전) 행이면? → 설립 N년차·연혁 모두 "연혁 준비 중"으로 폴백, 크래시 금지.
- `milestones` jsonb가 예상 스키마와 다르면(파싱 실패)? → 방어적 파서가 빈 연혁으로 폴백, 다른 지표 표시에는 영향 없음.
- 신규 2지표 도입으로 기존 순위가 크게 흔들리는 경우? → 허용된 변화(가중치 재조정은 명시적으로 승인된 변경, FR-011). 급격한 순위 변동 자체를 막는 별도 완충 장치는 두지 않는다.
- drag가 이론상 음수로 계산되는 경우(방어적)? → 0으로 clamp(마찰비용은 음수가 될 수 없으므로 데이터 이상으로 간주).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 저비용 지표를 산출해야 한다. `drag = Σ마찰비용(수수료+세금) / (초기평가액 + 누적 입금액)` 이며, 점수 `= clamp01(1 - drag/0.02) × 100`. `drag`를 산출할 수 없는 경우(투입원금 0 등) 점수는 50점으로 고정하고 "데이터 부족"을 표시해야 한다.
- **FR-002**: 시스템은 저레버리지 지표를 산출해야 한다. `부채비율 = 부채 / 자산평가액` 이며, 점수 `= clamp01(1 - 부채비율/0.4) × 100`(무차입은 항상 100점). 부채가 0보다 큰데 현재 자산 평가액(시세) 조회에 실패한 경우 점수는 50점으로 고정하고 "데이터 부족"을 표시해야 한다.
- **FR-003**: 총점 가중치는 다음 표를 따라야 한다(기존 5개 지표는 상대 비중을 유지한 채 ×0.8로 축소하고, 신규 2개 지표가 나머지 20%를 채운다):

  | 지표 | 가중치 |
  |------|--------|
  | 보유기간 가중 수익률 | 24% |
  | 역발상 매수율 | 20% |
  | 시장 대비 성과 | 16% |
  | 분산도 일관성 | 12% |
  | 적립 일관성 | 8% |
  | 저레버리지 | 10% |
  | 저비용 | 10% |
  | 합계 | 100% |

- **FR-004**: 랭킹 점수·순위 산정에 자산 금액과 절대 수익률(XIRR 절대값)을 지표로 추가해서는 안 된다(개요의 명시적 결정 참조). 시장 국면 대비 상대 성과는 기존 "시장 대비 성과" 지표만으로 반영한다.
- **FR-005**: `ranking_scores`에 `score_version` 컬럼을 도입해 이 기능 이후 계산된 행은 `2`로 표기해야 한다. 이전 방식(5지표)으로 계산되어 `score_version`이 `1` 또는 `null`인 행은 리더보드에서 배제하지 않고 그대로 표시해야 한다.
- **FR-006**: `score_version`이 `1`(또는 `null`)인 행은 별도의 배치·수동 마이그레이션 없이, 그 행의 소유자가 `/dashboard` 또는 `/ranking`을 방문할 때 자연스럽게 7지표(`score_version=2`)로 재계산·저장되어야 한다.
- **FR-007**: 리더보드의 각 행은 클릭(탭) 가능해야 하며, 탭 시 해당 지주회사의 프로필 바텀시트를 열어야 한다.
- **FR-008**: 프로필 바텀시트는 다음을 표시해야 한다: 지주회사명, 등급 배지, 총점, 7개 지표 각각의 점수·가중치 막대, 설립 후 경과 연차(N년차), 회사 연혁(설립 → 첫 매수 → 첫 해외투자 → 첫 배당 → 자본배분 계획 완수 ×N → 드로다운 인내 통과, 날짜순 나열, 날짜만 표기).
- **FR-009**: 랭킹 테이블과 프로필 바텀시트는 자산 금액, 보유 종목명, XIRR 절대값을 어떤 형태로도 저장하거나 노출해서는 안 된다(비공개 불변식). 이는 이 기능이 존재하는 한 항상 지켜져야 하는 구조적 제약이다.
- **FR-010**: 프로필 바텀시트는 다음 폴백을 지원해야 한다 — 해당 지표 점수가 `null`이거나 `score_version=1`이면 그 지표를 "산출 대기"로 표시, `milestones`가 없으면 연혁 영역을 "연혁 준비 중"으로 표시.
- **FR-011**: 033 스펙 FR-013이 명시한 "랭킹 채점 변경 금지" 잠금은 이 기능(034)으로 명시적으로 해제한다. 저비용·저레버리지 지표 신설(FR-001, FR-002)과 그로 인한 가중치 재조정(FR-003)은 이 스펙에서 승인된 변경이다. 033 FR-013의 다른 항목(연환산 90일 게이트, 복리 무중단 계산, CFO 코멘트)에 대한 잠금은 계속 유효하며 이 기능으로 영향받지 않는다.
- **FR-012**: `milestones` jsonb는 아래 "Key Entities"에 정의된 v1 스키마를 따라야 하며, 투입자본 금액이나 종목명 라벨을 포함해서는 안 된다(날짜만 저장). 033의 게이미피케이션 헌법(시장 결과 비노출) 정합성을 유지해야 한다.

### Key Entities

- **랭킹 점수 행(`ranking_scores`)**: 032의 5지표 컬럼에 더해 `leverage_score`, `cost_score`(둘 다 nullable, 구버전은 null), `score_version`(정수, 기본 1), `founded_at`(날짜), `milestones`(jsonb)가 추가된다.
- **공개 마일스톤(`milestones` jsonb, v1 스키마)**:
  ```jsonc
  {
    "v": 1,
    "plans_completed": 2,
    "plan_completed_dates": ["2026-03-02"],
    "drawdowns_passed": [{ "bucket": 20, "recovered_at": "2025-11-03" }],
    "first_buy_at": "2024-01-15",
    "first_overseas_at": "2024-05-02",
    "first_dividend_at": "2024-03-10"
  }
  ```
  투입자본 돌파 항목(금액 노출)과 종목명 라벨은 스키마에 존재하지 않는다. 설립 주년은 이 jsonb가 아니라 `founded_at`에서 매 조회 시 파생한다.
- **비공개 불변식**: 자산 금액·보유 종목명·XIRR 절대값은 `ranking_scores` 테이블과 프로필 바텀시트 어디에도 저장·노출되지 않는다. 이 불변식은 스키마 설계(수치가 애초에 존재하지 않는 컬럼 구성)로 구조적으로 보장되어야 하며, 화면단 필터링에 의존해서는 안 된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 무차입 지주회사는 100%의 경우 저레버리지 지표 100점으로 산출된다.
- **SC-002**: `drag` 산출 불가 또는 부채>0인데 시세 조회 실패인 모든 케이스에서 해당 지표가 크래시 없이 50점 + "데이터 부족"으로 표시된다.
- **SC-003**: 리더보드에서 임의로 연 프로필 바텀시트 표본 전수에서 자산 금액·종목명·XIRR 절대값 노출 0건.
- **SC-004**: `score_version=1` 행을 가진 계정이 본인 대시보드 또는 랭킹을 1회 방문한 뒤 `score_version=2`로 전환되는 비율 100%.
- **SC-005**: 구버전(`score_version=1`) 행이 리더보드 표시에서 배제되는 사례 0건(항상 그대로 노출).
- **SC-006**: 7지표 가중합이 항상 100(±반올림 오차)으로 산출된다.

## Assumptions

- `computeStyle`이 이미 계산하던 `drag`(마찰비용/투입원금)과 부채비율 산식은 그대로 재사용하며, 값 자체의 정의는 이 스펙에서 바꾸지 않는다(호출 경로만 랭킹에 배선).
- 부채 데이터는 기존 부채(`loadLiabilities`) 인프라를 재사용하며 신규 부채 개념을 도입하지 않는다.
- 드로다운 인내 통과 이력은 033에서 정의한 `loadDrawdownEpisodes()`의 "통과" 판정을 그대로 재사용한다(033의 판정 기준을 034가 재정의하지 않음).
- 연혁 원천(`journeyMilestones()` 등 기존 마일스톤 계산)이 이미 보유한 날짜 정보를 재사용하며, 034는 이를 "공개 가능한 형태"로 요약해 jsonb에 저장하는 책임만 진다.
- 리더보드는 이미 전체 유저 행을 조회하고 있어(032), 프로필 바텀시트를 위한 추가 API 호출은 필요하지 않다(같은 조회 결과를 재사용).
- 033 게이미피케이션 헌법(시장 결과 비노출·스타일 중립)은 이 기능에도 상위 규범으로 적용된다.

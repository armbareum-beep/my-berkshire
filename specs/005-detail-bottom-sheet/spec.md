# Feature Specification: 상세 바텀시트(드롭시트) — 체감 속도 개선

**Feature Branch**: `005-detail-bottom-sheet`
**Created**: 2026-06-22
**Status**: Draft
**Input**: User description: "화면에서 상세페이지로 이동하는 버튼 누르면 드롭시트가올라와서 X누르면 없어지게 할 수도있어? 그럼 좀 더 빠르게 느껴질까?"

---

## 배경 및 맥락

현재(웹):
- 목록(보유 자산, 검색 결과, 대시보드 카드)에서 종목·지수를 탭하면 `/stocks/[symbol]` 또는 `/index/[symbol]` **전체 페이지로 라우트 이동**한다.
- 전체 페이지 이동은 (1) 흰 화면 전환·로딩 체감, (2) 돌아올 때 직전 목록 스크롤 위치·필터 상태 상실, (3) "잠깐 보고 닫기"에 과한 무게감이라는 문제가 있다.

가설(사용자 질문):
> 상세로 가는 버튼을 누르면 전체 페이지 대신 **화면 아래에서 시트가 올라오고(드롭시트)**, **X로 닫으면** 원래 목록으로 즉시 돌아간다. 이렇게 하면 "더 빠르게 느껴질" 것이다.

이 기능은 새로운 데이터를 만들지 않는다. 기존 정보를 **다른 표현 방식(오버레이 시트)** 으로 보여주어 **체감 속도와 맥락 유지**를 개선하는 UX 변경이다.

이 기능의 1차 대상은 **종목/지수 상세 한 화면이 아니라, 홈(대시보드)에 있는 여러 섹션 카드·배너**다. 홈에는 리포트·투시(사업부 실적)·공시·순자산·연혁·보유·배당·리밸런싱·거래기록·가져오기 등 서로 다른 페이지로 가는 섹션이 많다. 이들 중 "보기만 하고 닫는" 흐름에 맞는 **정보 조회형** 섹션에 바텀시트를 적용한다.

## Clarifications

### Session 2026-06-22

- Q: 홈 화면의 어떤 섹션 이동을 바텀시트로 열까? → A: **정보 조회형 섹션만** — 리포트·투시(사업부 실적)·공시·순자산·연혁·보유 현황 조회, 그리고 종목/지수 상세. 거래기록·리밸런싱·가져오기·계좌관리처럼 **입력·편집·다단계 작업** 페이지는 바텀시트 대상이 아니며 기존 전체 페이지 이동을 유지한다.
- Q(구현 중 발견): `/company`(홈 헤더 회사명 ›)는 "연혁" 조회가 아니라 회사명 변경·전환·삭제·신규설립 **관리/편집 페이지**였음 → A: **작업형으로 확정**(전체 페이지 유지, 시트 제외). "연혁=조회형"은 별도의 연혁 화면이 생길 때 재검토.
- Q(구현 중 결정): 시트의 "전체 보기" 버튼(FR-007) 필요한가? → A: **제거**. 시트가 전체 내용을 담고 URL이 이미 전체 페이지 주소라 공유·딥링크는 그대로 되고, 같은 내용 재이동이라 실익이 없음.
- 추가 시트(사용자 요청): `/cash`(현금)·`/style`(운용 스타일)도 조회형 시트로 추가. 별도 버그 수정: 하단 탭바가 `page-in` transform 잔류로 콘텐츠 하단에 갇히던 것 → page-in을 opacity-only로 변경해 뷰포트 고정 복구.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 홈의 조회형 섹션을 시트로 즉시 확인 (Priority: P1)

사용자가 홈(대시보드)에서 정보 조회형 섹션 카드/배너(예: 분기 리포트, 사업부 실적(투시), 공시 소식, 순자산, 회사 연혁)를 탭하면, 페이지가 전환되는 대신 화면 아래에서 시트가 올라와 그 섹션 내용을 보여준다. 사용자는 오른쪽 위 X(또는 아래로 스와이프, 시트 밖 배경 탭)로 닫고, 닫으면 홈이 **스크롤 위치 그대로** 다시 보인다. 그래서 여러 섹션을 "탭 → 보고 → 닫고 → 다음 섹션"으로 빠르게 훑을 수 있다.

**Why this priority**: 홈은 여러 섹션을 오가며 점검하는 콘솔이다. 섹션마다 전체 페이지로 갔다 뒤로 돌아오는 왕복이 가장 잦은 마찰이고, 이를 시트로 줄이는 것이 이 기능의 핵심 가치다.

**Independent Test**: 홈에서 리포트(또는 공시·순자산) 섹션을 탭해 시트가 올라오고, X로 닫았을 때 같은 스크롤 위치의 홈으로 돌아오는지 확인.

**Acceptance Scenarios**:

1. **Given** 홈을 일부 스크롤한 상태, **When** 조회형 섹션 카드(예: 분기 리포트)를 탭, **Then** 페이지 이동 없이 화면 하단에서 시트가 올라와 해당 섹션 내용이 표시된다.
2. **Given** 시트가 열린 상태, **When** X를 탭, **Then** 시트가 닫히고 홈이 **같은 스크롤 위치**로 보인다.
3. **Given** 시트가 열린 상태, **When** 시트 밖 어두운 배경을 탭하거나 시트를 아래로 스와이프, **Then** X와 동일하게 닫힌다.
4. **Given** 시트가 열린 상태, **When** 기기 뒤로 가기(브라우저 back), **Then** 다른 페이지로 나가지 않고 시트만 닫힌다.
5. **Given** 시트 콘텐츠가 화면보다 길다, **When** 시트 내부를 스크롤, **Then** 뒤 홈은 움직이지 않고 시트 내용만 스크롤된다.

### User Story 2 — 종목·지수 상세를 시트로 미리보기 (Priority: P1)

사용자가 보유 목록·구성·검색 결과에서 종목 또는 지수를 탭하면 동일하게 상세 시트가 올라온다. 닫으면 직전 목록(또는 검색어·결과)이 그대로 유지되어 연속으로 여러 종목을 빠르게 비교·확인할 수 있다.

**Why this priority**: 종목 점검과 검색 탐색은 가장 빈번한 흐름이며, US1과 동일한 시트 메커니즘을 재사용한다. 조회형 섹션과 함께 일관된 "보고 닫기" 경험을 완성한다.

**Independent Test**: 보유 목록 또는 검색 결과에서 한 종목을 탭 → 상세 시트 표시 → 닫으면 직전 목록/검색 상태가 유지되는지 확인.

**Acceptance Scenarios**:

1. **Given** 보유 목록을 일부 스크롤한 상태, **When** 종목 줄을 탭, **Then** 화면 하단에서 상세 시트가 올라오고, 닫으면 같은 스크롤 위치로 돌아온다.
2. **Given** "코스피"로 검색해 결과가 보이는 상태, **When** 결과 항목을 탭, **Then** 검색 화면 위로 상세 시트가 올라온다.
3. **Given** 시트를 닫음, **When** 닫기 완료, **Then** 검색어와 결과 목록이 그대로 남아 다음 항목을 바로 탭할 수 있다.

---

### User Story 3 — 입력·작업 페이지는 시트로 가두지 않음 (Priority: P1)

사용자가 거래 기록·리밸런싱 계획·거래내역 가져오기·계좌 관리처럼 입력·편집·다단계 작업이 필요한 섹션을 탭하면, 시트가 아니라 기존처럼 전체 페이지로 이동한다. 이런 작업은 "잠깐 보고 닫기" 흐름이 아니라 집중이 필요하기 때문이다.

**Why this priority**: 시트의 범위를 명확히 가르는 경계 규칙이다. 이 구분이 없으면 입력 작업이 좁은 시트에 갇혀 오히려 사용성이 나빠진다. 범위 오인을 막아 받아들임 테스트의 기준이 된다.

**Independent Test**: 홈에서 "거래내역 가져오기"(또는 리밸런싱 배너)를 탭하면 시트가 아니라 해당 전체 페이지로 이동하는지 확인.

**Acceptance Scenarios**:

1. **Given** 홈, **When** 입력·작업 섹션(거래기록·리밸런싱·가져오기·계좌관리)을 탭, **Then** 바텀시트가 아니라 기존 전체 페이지로 이동한다.
2. **Given** 조회형 시트가 열린 상태에서 그 안에 작업 진입 링크가 있다, **When** 해당 링크를 탭, **Then** 시트가 닫히고 작업 전체 페이지로 이동한다.

---

### User Story 4 — 시트에서 전체 페이지로 확장 & 공유 (Priority: P3)

시트는 빠른 확인용이지만, 사용자가 더 깊게 보거나 링크를 공유하고 싶을 때 시트 안의 "전체 보기"로 기존 상세 페이지(`/stocks/[symbol]`·`/index/[symbol]`)로 이동할 수 있다. 외부에서 들어온 딥링크/북마크 URL은 기존처럼 전체 페이지로 열린다.

**Why this priority**: 시트는 기존 상세 페이지를 대체하는 게 아니라 빠른 진입점이다. 공유·딥링크·SEO 경로를 깨지 않도록 전체 페이지를 보존한다.

**Independent Test**: 시트의 "전체 보기"가 해당 종목의 전체 페이지로 이동하고, 그 URL을 직접 열면 동일 페이지가 뜨는지 확인.

**Acceptance Scenarios**:

1. **Given** 종목 상세 시트, **When** "전체 보기" 탭, **Then** 해당 종목의 전체 상세 페이지로 이동한다.
2. **Given** 외부에서 `/stocks/AAPL` 링크로 진입, **When** 페이지 로딩, **Then** 시트가 아닌 기존 전체 페이지가 표시된다.

---

### Edge Cases

- 시트가 열린 상태에서 또 다른 종목을 탭하면? → 현재 시트가 새 종목 내용으로 교체된다(시트를 닫았다 다시 열지 않음).
- 상세 데이터 로딩 중/실패 시? → 시트는 즉시 올라오고 내부에 로딩 표시, 실패 시 기존과 동일한 오류 문구를 시트 안에 표시(시트 자체는 정상 동작).
- 데이터가 막혀 있던 항목(예: "곧 공개됩니다" ETF/지수)을 탭하면? → 기존과 동일한 안내를 시트 안에 표시하거나, 진입이 막혀 있으면 시트를 열지 않는다(기존 차단 규칙 유지).
- 시트 열림 중 화면 회전/리사이즈? → 시트는 열린 상태와 표시 종목을 유지한다.
- 가로가 넓은 화면(데스크톱)에서는? → 화면 크기와 무관하게 동일한 하단 시트로 통일한다(별도 모달/전체 페이지 분기 없음).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: **정보 조회형** 섹션(분기/연간 리포트, 사업부 실적(투시), 공시 소식, 순자산, 회사 연혁, 보유 현황 조회)과 종목/지수 상세의 진입 동작은 전체 페이지 라우팅 대신 화면 하단에서 올라오는 시트를 열어야 한다.
- **FR-001a**: **입력·편집·다단계 작업** 페이지(거래 기록, 리밸런싱 계획, 거래내역 가져오기, 계좌 관리)는 바텀시트 대상이 아니며, 탭하면 기존처럼 전체 페이지로 이동해야 한다.
- **FR-001b**: "조회형 vs 작업형"의 분류 기준은 **"보고 닫는 흐름인가(조회형) vs 입력·편집·여러 단계가 필요한가(작업형)"** 이며, 새 섹션이 추가될 때도 이 기준으로 판정한다.
- **FR-002**: 시트는 X 버튼, 시트 밖 배경 탭, 아래로 스와이프, 기기/브라우저 뒤로 가기 네 가지 방법으로 닫을 수 있어야 한다.
- **FR-003**: 시트를 닫으면 시트를 연 직전의 목록 화면이 **스크롤 위치·검색어·필터 등 상태를 보존**한 채 다시 보여야 한다.
- **FR-004**: 시트가 열려 있는 동안 뒤 배경 목록은 스크롤/조작되지 않아야 하며(스크롤 락), 시트 내부 콘텐츠만 스크롤되어야 한다.
- **FR-005**: 시트는 기존 상세 화면의 **전체 내용**을 그대로 담아야 한다(핵심 요약본이 아니라 동일 정보 전부). 화면보다 길면 시트 내부 스크롤로 모두 볼 수 있어야 한다.
- **FR-005a**: 시트는 모바일·데스크톱·와이드 화면 모두에서 동일한 하단 시트 형태로 표시되어야 한다(화면 크기에 따라 모달/전체 페이지로 분기하지 않는다).
- **FR-006**: 시트는 종목과 지수 두 종류의 상세를 모두 표시할 수 있어야 한다.
- **FR-007**: ~~시트 안에서 기존 전체 상세 페이지로 이동하는 "전체 보기" 경로를 제공해야 한다.~~ **철회(구현 중 결정)** — 시트가 상세 **전체 내용**을 담고(FR-005) 시트의 URL이 이미 전체 페이지 주소라 공유·딥링크·새로고침이 그대로 동작(FR-008). "전체 보기" 버튼은 같은 내용으로의 중복 이동이라 제거함. ("시트=요약" 가정의 잔재였음)
- **FR-008**: 외부 딥링크/북마크/공유 URL(`/stocks/[symbol]`, `/index/[symbol]`)은 기존처럼 전체 페이지로 열려야 한다(시트는 앱 내부 탐색의 진입점일 뿐 URL 계약을 대체하지 않는다).
- **FR-009**: 상세 데이터 로딩 지연·실패가 시트 자체의 열림/닫힘 동작을 막아서는 안 되며, 오류는 시트 내부에 기존과 동일한 안내로 표시되어야 한다.
- **FR-010**: 시트가 열린 상태에서 다른 항목을 선택하면 같은 시트의 내용이 교체되어야 한다.
- **FR-011**: 시트의 열림/닫힘 전환에는 즉각적이고 부드러운 시각적 모션(아래에서 위로 슬라이드)이 있어야 한다.

### Key Entities

*해당 없음 — 새 데이터 엔티티를 도입하지 않는다. 표시 대상은 기존 종목·지수 상세 정보다.*

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 조회형 섹션·종목 탭 후 시트가 **0.3초 이내**에 화면에 나타나기 시작한다(콘텐츠 로딩 완료 전이라도 시트 프레임은 즉시 표시).
- **SC-002**: 시트를 닫은 뒤 직전 화면(홈 또는 목록)이 **추가 로딩/리렌더 체감 없이** 같은 스크롤 위치로 복원된다(재요청 0회).
- **SC-003**: 홈에서 "섹션 탭 → 확인 → 닫기 → 다음 섹션 탭"을 연속 5회 수행할 때, 기존 전체 페이지 왕복 대비 **총 소요 시간이 단축**된다(사용성 테스트로 검증).
- **SC-004**: 사용자 인지 조사에서 다수 사용자가 시트 방식을 기존 페이지 이동보다 "더 빠르게 느껴진다"고 응답한다.
- **SC-005**: 시트 닫기 4가지 방법(X·배경탭·스와이프·뒤로가기) 모두에서 의도치 않게 앱 밖/다른 페이지로 빠지는 일이 0건이다.

## Assumptions

- **진입점 범위(확정)**: 홈(대시보드)의 **정보 조회형 섹션**(리포트·투시·공시·순자산·연혁·보유 현황)과 종목/지수 상세가 대상이다. 입력·작업 페이지(거래기록·리밸런싱·가져오기·계좌관리)는 제외하고 전체 페이지 이동을 유지한다(US3·FR-001a).
- **조회형/작업형 경계 사례**: 배당 일정 "조회"는 조회형(시트), 계좌 "관리/편집"은 작업형(전체 페이지)으로 분류한다. 보유 현황은 조회형이나, 그 안에서 종목을 고르면 종목 상세 시트로 이어진다.
- **표시 깊이(확정)**: 시트는 기존 상세 화면의 전체 내용을 담는다. "전체 보기"는 콘텐츠 축약 때문이 아니라 공유·딥링크용 진입점으로 유지한다.
- **플랫폼(확정)**: 모바일·데스크톱·와이드 화면 모두 동일한 하단 시트로 통일한다.
- **데이터/차단 규칙 보존**: 기존 상세의 데이터 소스·오류 문구·진입 차단("곧 공개됩니다") 규칙은 그대로 재사용한다.
- **URL 계약 보존**: 기존 `/stocks/[symbol]`·`/index/[symbol]` 라우트와 그 동작은 유지된다.

# Feature Specification: 부동산 사업부 (Real Estate Division)

**Feature Branch**: `011-real-estate-division`  
**Created**: 2026-06-24  
**Status**: Draft  
**Input**: User description: "부동산 등 수기자산을 '수익 내는 사업부'로 승격. 임대수익(실현)은 사업부 자체 원장에 자산별 기록(주식 events 안 건드림), 평가차익(미실현)은 수기 평가가−취득가에 출처·갱신일 표기. 시세 자동연동 없이 전부 수기. 홈 재배치: 보유계좌 → 주식 자산구성 → 부동산 사업부 → 현금. 매입가/현재가 입력 안전장치."

## 배경 (Why)

지금까지 부동산 등 수기자산은 "그냥 순자산"으로만 잡혀, 취득가 대비 평가차익(미실현)만 누적수익률에 합산됐다(010). 하지만 버크셔식 지주회사 메타포에서 부동산은 주식과 동급의 **수익 내는 사업부**다 — 임대료라는 실현 수익을 낳고, 가치도 변한다.

이 기능은 부동산을 **자체 수익 원장을 가진 사업부**로 승격한다:
- **임대수익(실현)** — 자산별로 받은 임대료를 기록. 주식의 배당·매매차익에 대응.
- **평가차익(미실현)** — 수기 평가가 − 취득가. 단 평가는 추정이므로 **출처·갱신일**을 박아 정직성을 지킨다.

이로써 주식 사업부(시세·검증)와 부동산 사업부(수기·추정)가 각자 기준으로 실현/미실현을 보고하고, 홈은 "사업부별 자산"으로 읽힌다. **측정 기준이 다른 두 사업부의 수익을 한 숫자로 섞지 않는 것**이 핵심 원칙이다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 홈에서 부동산을 하나의 사업부로 본다 (Priority: P1)

사용자는 홈 화면에서 자산이 **사업부별**로 정렬된 것을 본다: 보유계좌(주식) → 주식 자산구성 → **부동산 사업부** → 현금(맨 끝). 부동산 사업부 카드는 보유 부동산의 현재 평가액과 **평가차익(미실현) 수익률**을 보여준다.

**Why this priority**: 부동산이 "순자산 한 줄"이 아니라 독립 사업부로 보이는 것 자체가 이 기능의 골격. 임대수익이 없어도(평가차익만으로) MVP로 성립한다.

**Independent Test**: 취득가·현재가가 입력된 부동산이 있는 장부에서, 홈에 부동산 사업부 카드가 지정된 순서로 뜨고 평가차익 수익률이 사실과 일치하는지 확인.

**Acceptance Scenarios**:

1. **Given** 취득가 12.5억·현재가 18억 부동산이 있을 때, **When** 홈을 연다, **Then** 부동산 사업부 카드에 평가차익 +5.5억(+44%)이 표시된다.
2. **Given** 홈 자산 섹션을 볼 때, **When** 화면을 본다, **Then** 순서가 보유계좌 → 주식 자산구성 → 부동산 사업부 → 현금 이다.
3. **Given** 부동산 자산이 하나도 없을 때, **When** 홈을 연다, **Then** 부동산 사업부 카드는 표시되지 않는다(빈 사업부 강요 안 함).

---

### User Story 2 - 임대수익을 자산별로 기록하고 실현수익을 본다 (Priority: P2)

사용자는 특정 부동산에 받은 **임대료를 날짜·금액으로 기록**한다. 그 자산(과 부동산 사업부)의 **실현수익**으로 누적되어, 평가차익(미실현)과 분리되어 표시된다.

**Why this priority**: "수익 내는 사업부"의 핵심. 다만 P1(평가차익) 위에 얹히는 증분이라 두 번째.

**Independent Test**: 한 부동산에 임대료 여러 건을 기록하면, 그 자산의 실현수익 합과 사업부 실현수익이 정확히 누적되고, 평가차익과 별도로 표시되는지 확인.

**Acceptance Scenarios**:

1. **Given** 취득가 12.5억 부동산에 월 임대 500만을 12건 기록했을 때, **When** 부동산 사업부를 본다, **Then** 실현(임대) 누적 6,000만이 표시되고, 평가차익(미실현)과 분리되어 보인다.
2. **Given** 임대수익을 기록할 때, **When** 저장한다, **Then** 그 금액이 **주식 투자원금·복리 무중단·XIRR에는 전혀 영향을 주지 않는다**(자체 원장).
3. **Given** 임대수익·평가차익이 모두 있을 때, **When** 부동산 사업부 종합수익률을 본다, **Then** (평가차익 + 누적임대) ÷ 취득가 로 계산된다.

---

### User Story 3 - 정직한 평가: 출처·갱신일·입력 안전장치 (Priority: P3)

사용자가 부동산 평가액을 입력/수정할 때 **평가 출처(KB시세·실거래가·감정가 등)와 갱신일**을 남긴다. 또 매입가·현재가를 헷갈려 입력해 평가손실이 나면 **확인을 받는다**.

**Why this priority**: 추정 수치의 신뢰를 지키는 정직성 레이어(헌장 II). 핵심 흐름은 P1/P2로 돌아가므로 마지막.

**Independent Test**: 평가 입력 화면에서 출처·갱신일이 저장·표시되는지, 현재가 < 매입가 입력 시 확인 안내가 뜨는지 확인.

**Acceptance Scenarios**:

1. **Given** 부동산 평가액을 입력할 때, **When** 출처·평가일을 함께 입력한다, **Then** 사업부 카드/상세에 "추정 · {출처} · {평가일}"이 표시된다.
2. **Given** 현재가를 매입가보다 낮게 입력할 때(예: 매입가 18억, 현재가 12.5억), **When** 저장을 시도한다, **Then** "평가손실이 맞나요? 매입가/현재가가 바뀌지 않았는지 확인하세요" 안내를 받는다.

---

### User Story 4 - 부동산을 팔아 매도차익을 실현한다 (Priority: P2)

사용자는 보유 부동산을 **매도**한다(매도가·매도일). 그 자산은 현재 보유에서 빠지고(미실현 평가차익 해소), **매도차익이 실현**으로 잡힌다. 매도된 자산은 이력으로 남아 "이 사업부가 과거에 이만큼 벌었다"가 보존된다.

**Why this priority**: 실현 그림을 완성하는 조각(주식의 매도차익에 대응). 매도 없이도 P1·P2(보유·임대)는 돌아가므로 P2.

**Independent Test**: 보유 부동산을 매도 기록하면, 미실현에서 빠지고 매도차익이 실현 누적에 더해지며, 사업부 종합수익률이 비용 차감 후로 맞는지 확인.

**Acceptance Scenarios**:

1. **Given** 취득가 12.5억 부동산을 18억에 매도(비용 5천만)했을 때, **When** 부동산 사업부를 본다, **Then** 매도차익 5억(= 18억 − 12.5억 − 0.5억)이 **실현**으로 잡히고, 그 자산은 보유(미실현)에서 사라진다.
2. **Given** 매도된 자산일 때, **When** 사업부 이력을 본다, **Then** 매도일·매도가·매도차익이 이력으로 남는다.
3. **Given** 매도 대금을 기록할 때, **When** 저장한다, **Then** 그 현금은 주식 events·투입원금·XIRR에 자동 연동되지 않는다(재투자 시 사용자가 따로 증자 기록).

---

### Edge Cases

- **취득가 없는 자산**: 평가차익·실현수익률을 계산할 수 없으므로 **수익률 스코프에서 제외**(가치는 순자산·총자산에 여전히 합산). 수익률 자리엔 "취득가 입력 시 수익률 표시".
- **임대수익만 있고 평가차익 0**: 종합수익률 = 실현(임대)분만.
- **여러 부동산**: 사업부 종합 = 자산별 합산(취득가 있는 것만).
- **임대수익 재투자**: 받은 임대를 주식에 넣으면 그건 별도의 증자(DEPOSIT)로 기록 — 부동산 임대 기록과 이중계상되지 않음.
- **표시 통화**: 부동산도 기능통화 KRW 기준, ₩/$ 토글 시 환산.
- **매도**: 보유 전체 매도를 기본으로 본다(부분 매도는 범위 밖). 매도 후 자산은 미실현에서 빠지고 실현 이력으로만 남는다.
- **세금·비용 입력**: 항목(취득세·양도세·중개료·재산세 등)을 쪼개 묻지 않는다 — 거래당 한 칸. 모르면 0(비용 미반영, 가짜 추정 금지).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 부동산 등 수기자산을 **사업부**로 다루어, 평가차익(미실현)과 임대수익(실현)을 **분리**해 보여줘야 한다.
- **FR-002**: 시스템은 **임대수익을 자산별로(날짜·금액) 기록**할 수 있어야 하며, 이는 부동산 사업부의 **자체 수익 원장**에 저장된다.
- **FR-003**: 시스템은 임대수익이 **주식 투자 원장(events)·투입원금·복리 무중단·XIRR에 영향을 주지 않도록** 완전히 분리해야 한다(이중계상 금지, 헌장 V).
- **FR-004**: 시스템은 부동산 사업부의 **실현수익**(임대 + 매도차익, 비용 차감 후)과 **평가차익(미실현)**(현재 추정가 − 실질취득가)을 각각, 그리고 **종합**으로 제시해야 한다. 모든 수익률의 분모는 **실질취득가**(취득가 + 취득 부대비용)다.
- **FR-005**: 시스템은 부동산 평가가 **수기·추정**임을 명시하고, **평가 출처와 갱신일**을 입력·표시해야 한다. 시세 자동연동은 하지 않는다.
- **FR-006**: 시스템은 현재가가 매입가보다 낮게 입력될 때 **확인 안내**를 제공해야 한다(입력 실수 방지).
- **FR-007**: 시스템은 홈 자산 섹션을 **보유계좌 → 주식 자산구성 → 부동산 사업부 → 현금** 순으로 배치해야 한다.
- **FR-008**: 시스템은 부동산 자산이 없으면 부동산 사업부 카드를 **표시하지 않아야** 한다(빈 사업부 강요 금지, 헌장 II).
- **FR-009**: 시스템은 **취득가 없는 수기자산**을 수익률 계산에서 제외하되, 그 가치는 순자산·총자산에 합산해야 한다.
- **FR-010**: 부동산 사업부의 실현+미실현 합산이 **총자산 누적수익률**(히어로)과 **사업부별 카드**(순자산 페이지)에 반영되어야 한다. 단 **연복리(XIRR)는 여전히 주식 사업부 전용**이다.
- **FR-011**: 모든 부동산 금액은 기능통화 KRW로 저장하고, 표시 통화 토글에 따라 환산한다.
- **FR-012**: 시스템은 부동산 **매도**(매도가·매도일·매도비용)를 기록할 수 있어야 하며, 매도 시 자산을 현재 보유(미실현)에서 제외하고 **매도차익(= 매도가 − 실질취득가 − 매도비용)을 실현**으로 누적, 매도 이력(매도일·매도가·차익)을 보존해야 한다.
- **FR-013**: 시스템은 각 거래(취득·임대·매도)의 **세금·비용을 항목별로 쪼개지 않고 거래당 단일 합산 금액**으로 입력받아야 한다(입력 단순). 취득 부대비용은 실질취득가에 가산, 임대·매도 비용은 해당 실현에서 차감하여, 부동산 사업부 수익률은 모두 **비용 차감 후(net)**로 계산한다.

### Key Entities *(include if feature involves data)*

- **부동산 사업부 자산(Manual Asset)**: 기존 수기자산을 확장 — 이름·종류·현재 평가액·취득가·취득일에 더해 **취득 부대비용(세금·비용 단일 합산)·평가 출처·평가 갱신일**, 그리고 매도 시 **매도 상태·매도가·매도일·매도비용**. 종류별 "사업부" 라벨(부동산/토지/상가/벤처/대체).
- **임대수익 기록(Rental Income Entry)**: 부동산 사업부 자산별 실현 수익 — 자산 참조·날짜·금액(₩)·**비용(세금·관리비 등 단일 합산)**. 주식 `events`와 분리된 별도 원장(이중계상 방지).
- **부동산 사업부 수익(Division Return, 파생)**: 자산별·사업부 합산의 실현(임대 + 매도차익, 비용 차감 후)·미실현(평가차익)·종합 수익률. 분모는 실질취득가. 모두 위 원장에서 결정적으로 계산되는 파생값.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 취득가·현재가가 입력된 부동산의 평가차익 수익률이 실제 (현재가−취득가)/취득가와 100% 일치한다.
- **SC-002**: 임대수익을 기록해도 주식 XIRR·투입원금·복리 무중단 숫자가 **전혀 변하지 않는다**(완전 분리 검증).
- **SC-003**: 홈 자산 섹션 순서가 보유계좌 → 주식 자산구성 → 부동산 사업부 → 현금 으로 표시된다.
- **SC-004**: 부동산 평가에 출처·갱신일이 없으면 "추정" 외 출처 표기가 비어 있고, 입력하면 카드/상세에 함께 표시된다.
- **SC-005**: 현재가 < 매입가 입력 시 사용자가 저장 전에 확인 안내를 받는다(실수 입력이 조용히 마이너스로 통과하지 않는다).
- **SC-006**: 부동산을 매도하면 해당 자산이 미실현(보유)에서 사라지고 매도차익이 실현 누적에 더해지며, 매도 이력(매도일·매도가·차익)이 보존된다.
- **SC-007**: 세금·비용을 입력하면 부동산 사업부 수익률이 비용 차감 후(net)로 줄어들고, 입력은 거래당 한 칸으로 끝난다(항목별 분리 입력 없음).

## Assumptions

- 부동산 사업부 자산·임대수익은 기존 수기자산 서브시스템(주식 `events`와 분리)을 확장한다 — 단일 진실원천 원칙상 임대수익을 events에 넣지 않는다.
- 평가는 **전부 수기**다. 한국 KB시세는 공개 API가 없고, 국토부 실거래가 API는 아파트에 한정(상가·토지 제외)되어 부분적이라, 자동연동은 이번 범위에서 제외한다. (추후 아파트 한정 보조로만 재검토 가능.)
- "임대수익"은 상업용·주거용 부동산의 월세·전세 운용수익 등 실제 받은 현금을 의미하며, 받은 시점 기준으로 기록한다.
- 부동산 사업부 수익률의 분모는 **실질취득가**(취득가 + 취득 부대비용)다(주식의 투입원금에 대응). 취득가가 없으면 수익률 미산출.
- 010에서 만든 `manualAssetsCostBasis`·`computeBusinessReturns`·`BusinessReturnsCard`를 확장해 실현(임대)분을 합산한다.
- 연복리(XIRR)는 부동산에 적용하지 않는다(시세 시계열 없음). 부동산 사업부는 누적(실현/미실현) 관점만.
- 표시 통화·환산은 기존 규칙(₩ 저장, factor 환산)을 따른다.

## 구현 노트 (011 완료 — 설계 진화)

대화 중 "부동산 전용"에서 **사업부 일반 틀**로 확장:
- 종류를 사업부로 묶음(`computeDivisions`): 부동산(부동산·토지·상가) / **대체**(미술·수집, "대체자산") / 사업(비상장) / 기타. 수익 발생 사업부만 임대·배당 입력 노출.
- **코인·원자재는 주식에서 관리**(시세 있음 → events·XIRR). 수기 사업부 아님.
- 자산별 표시 = 자본수익률(가격상승). 사업부 = 실현(임대+매도차익)+미실현(평가차익).
- 화면: 홈 = 사업부 카드(전체 수익률 + 자산별 금액/번돈), 현금 최하단. /networth = 사업부별 비교 카드 + 보유목록(개별). 전용 페이지 `/real-estate` = 관리.

# Phase 0 Research: 복리 무중단 지표

대화로 합의된 설계 결정과 그 근거를 기록한다. 모든 NEEDS CLARIFICATION 해소됨.

## Decision 1: "복리 무중단"의 정의 — 끊김의 경계

**Decision**: 무중단 기간 = *소비성 자본인출(WITHDRAWAL)* 없이 자본을 복리 기계 안에서 굴려온 연속 기간. 끊김은 자본이 지주회사 밖으로 나가 소비될 때만.

**Rationale**: 멍거의 복리 제1원칙 — "필요 없이 절대 중단하지 마라". 복리를 중단시키는 것은 자본을 *빼서 쓰는 것*(차·집·생활비)이지 자본을 *어디에 두느냐*가 아니다. 소비성 인출만이 자본을 기계에서 영구히 제거한다.

**Alternatives considered**:
- *가동률/현금비중 기반 끊김* — 현금이 임계선 넘으면 끊김. **기각**: 현금도 복리 기계 안(이자·옵션·대기자본), 버핏式 dry powder를 처벌하는 자기모순. 임계선이 자의적이고 스타일 판단 위험.
- *순인출 누적이 음수일 때만* — 누적 입금 대비 누적 인출 추적. **기각**: 가족 장부에선 거의 항상 양수라 무의미하고, 한 번의 소비성 인출도 "중단"으로 봐야 교훈이 산다.

## Decision 2: 현금 보유는 끊김이 아니다

**Decision**: `cashWeight`(현금 비중)는 무중단 끊김 판정에 사용하지 않는다.

**Rationale**: 현금은 기계 밖으로 나간 게 아니다 — MMF·파킹·예금에서 이자가 붙고(복리), 기회 시 투입되는 대기 자본이다. 버핏의 현금 더미가 매년 이자를 버는 것과 같다. 현금 보유 = 자본배분 선택이지 복리 중단이 아니다.

**Alternatives considered**: 현금을 드래그로 보고 streak에 반영 — **기각**(Decision 1과 동일 근거, 스타일 중립·정직 원칙 위반 소지).

## Decision 3: 끊김 시 완전 리셋

**Decision**: 소비성 인출 발생 시 무중단 기간을 0에서 재시작(인출 시점 기준).

**Rationale**: "무중단"은 연속성 개념이다. 부분 감점은 의미가 모호하고, 0 리셋이 "방금 복리를 끊었다"는 명확한 교훈을 준다.

**Alternatives considered**: 인출액 비례 부분 차감 — **기각**(직관 불명확, 연속성 개념과 불일치).

## Decision 4: 표시 단위 (일/개월)

**Decision**: 일 단위로 계산하되 표시는 1개월 미만 "N일", 그 이상 "N개월". 첫날부터 노출.

**Rationale**: 90일 XIRR 공백을 메우는 게 목적(SC-001). 첫날 "무중단 1일"이라도 신호가 보여야 한다. 개월 환산은 외부 의존 없이 날짜 차로 계산.

**Alternatives considered**: 처음부터 개월만 — **기각**(초반 "0개월"은 신호 공백, 목적 위배).

## Decision 5: CAGR 사용자 노출 제거, 비율은 XIRR 단일

**Decision**: 사용자 화면의 CAGR 표시 제거(`PeriodReturns.tsx`, `returns/page.tsx`). XIRR을 "연 수익률" 등 평이한 라벨로 단일 노출. CAGR 계산 함수 자체는 내부 잔존 가능.

**Rationale**: CAGR은 추가 입금이 많을수록 부풀려져(입금으로 커진 것과 실력 구분 못 함) 적립 철학의 이 앱에선 오해를 부른다. 두 백분율 동시 노출이 혼동의 근원(SC-003).

**Alternatives considered**: CAGR 함수까지 삭제 — **기각**(내부 테스트·향후 분석 여지 보존, 최소 변경).

## Decision 6: 저장하지 않고 파생 계산

**Decision**: streak 상태를 DB에 저장하지 않고 `events`에서 매번 결정적으로 계산(`dashboard.ts` 조립 시).

**Rationale**: 단일 진실원천 원칙(헌장 V). O(n) 이벤트 1회 순회로 비용 무시 수준. 캐시/동기화 부담 없음.

**Alternatives considered**: streak 캐시 컬럼 — **기각**(이중 계상·정합 리스크, 불필요).

## Next.js 변형 주의

이 repo의 Next.js는 변형판이라 새 서버/클라이언트 경계나 라우팅 패턴을 도입할 때 `node_modules/next/dist/docs/`를 먼저 확인해야 한다. 다만 본 기능은 기존 컴포넌트(`cards.tsx`, `QuarterReportView.tsx`)에 표시 한 줄/블록을 더하고 순수 lib 함수를 추가하는 수준이라 새 Next API 도입은 없다 — 리스크 낮음.
